# CoDD（Coherence-Driven Development）解説

**想定読者**: SE、PM、SIerのマネージャー、AI駆動開発に興味があるエンジニア
**前提知識**: プログラミング経験あり。V字モデル・DI・ORM等の用語が分かる程度

---

## CoDDが解きたい問題

エンタープライズの巨大システム（数千万行級）で、**ある箇所を変更したとき、他のどこに影響が出るか？**

### なぜこれが難しいのか

普通のシステムなら、IDEの「参照を検索」やgrepで追える。でもエンタープライズのパッケージソフトだと、コード上の依存以外にこういう依存がある。

**コードを読めば追えるもの:**
- ファイルAがファイルBをimportしている
- 関数Aが関数Bを呼んでいる
- クラスAがクラスBを継承している

**コードを読んでも追えないもの:**
- 設定ファイルの値で、呼び出し先が実行時に変わる（Spring DIやプラグイン機構）
- マスターテーブルの値が、ビジネスロジックの分岐を制御している（コードにはSELECT文しかない）
- 「ここ変えたらあそこも変える」という暗黙のルール（どこにも書いてない。熟練者の頭の中だけ）

### 今どうしてるか

20年選手の熟練エンジニアが**頭の中のメンタルモデル**で解いている。「あ、ここ触ったらあのマスターが関係するな」「この設定だと本番ではStripeだけだから、PayPalは確認だけでいい」と直感で分かる。

### 何が問題か

その人が辞めたら、**誰もシステムを安全に触れなくなる**。若手は依存関係が見えないから、変え忘れて障害を起こす。

### CoDDが目指すこと

**熟練者のメンタルモデルの7割をAIで再現して、若手でも安全に改修できるようにする。**

完璧じゃなくていい。「ここ変えたらここも見ろ」が出るだけで激変する。

---

## CoDDの名前の意味

**CoDD = Coherence-Driven Development（変化点伝播駆動開発）**

### 従来のやり方（SDD = Spec-Driven Development）

```
要件定義 → 基本設計 → 詳細設計 → コード → テスト
```

片道切符。上から下にしか流れない。コード書いてて「この設計じゃ無理だ」となっても、要件定義書まで自動で戻らない。人間が気づいて手で直す。

### CoDDのやり方

```
要件定義 ⇄ 基本設計 ⇄ 詳細設計 ⇄ コード ⇄ テスト
```

双方向。どこを触っても、影響が上にも下にも自動で伝わる。

- コード直した → 詳細設計 → 基本設計 → 要件定義まで「ここも影響あるよ」が伝播
- 要件追加した → 基本設計 → 詳細設計 → コード → テストまで「ここ変えなきゃ」が伝播
- 途中の設計書を直した → 上にも下にも伝播

**仕様が駆動するのではない。設計の一貫性（Coherence）が駆動する。** だからCoherence-Driven。

---

## CoDDの思想: 何も発明しない

CoDDは新しい概念を何一つ発明していない。

ソフトウェア開発には数十年の歴史で確立されたベストプラクティスがある。V-model、UML（クラス図・シーケンス図・ER図・CRUD図）、依存管理、変更管理、回帰テスト計画。全部教科書に書いてある。正しいやり方は**みんな知っていた**。

でも実行できなかった。

- 詳細設計（クラス図・シーケンス図・ER図）を全ユースケース分書く → **数週間かかる**
- 書いた設計書を要件変更のたびに更新し続ける → **面倒だから放置される**
- 影響範囲を正確に追跡する → **熟練者の頭の中にしかない**
- 変更時に全成果物の整合性を保つ → **人間が手動で全部チェック**

知識の問題じゃなかった。**実行コストの問題だった。**

SIerが詳細設計を省略してきたのは、不要だからじゃない。人間がやるには高すぎたからだ。AIで実行コストがゼロに近づいた今、**正しい工程を全部やればいいだけ**。

CoDDがやっていること:

| やること | 元ネタ | CoDDでの使い方 |
|---------|-------|---------------|
| 設定マージ | Rails/nginx/ESLint | 2層設定アーキテクチャ |
| V-model工程 | ウォーターフォール | wave依存グラフ |
| 詳細設計（UML図） | V-model詳細設計 | AIが概要設計から自動生成 |
| 成果物間の依存管理 | Makefile（1976年） | 生成順序の自動制御 |
| フロントマター | Jekyll/Hugo | 成果物の自己記述メタ情報 |
| DAG | Airflow/Make | 依存グラフによる実行順序制御 |
| 訓練ループ | Deep Learning | 生成→レビュー→改善ループ |

世界中のエンジニアが独立に同じパターンに収束した = 正解の証拠。車輪の再発明をしていないことが、信頼性の根拠。

そして全ての成果物に**依存関係というメタ情報**を付ける。このメタ情報がChange Propagationに効く。要件が変わったら、依存グラフを辿って影響範囲を自動特定し、該当する成果物だけを再生成する。人間がやると漏れる。AIなら漏れない。疲れない。何度でもやり直せる。

**これは人間にはもともと無理ゲーだった。AIがやる仕事。**

### 一言で言えば

IPAの共通フレーム（SLCP）に書いてある開発工程を、省略せずに全部やる。技術リードやPMが本来やるべきことを、AIが全部やる。唯一の追加要素は、**全ての成果物に依存関係のメタ情報を付ける**こと。それだけ。

このメタ情報があるから、Change Propagationが動く。要件が変わったとき、依存グラフを辿って影響範囲を自動特定し、該当する成果物だけを再生成できる。メタ情報がなければただの成果物生成ツール。メタ情報があるからCoDD。

## CoDDの3本柱

### 柱1: 要件から設計を自動生成する

要件定義（機能要件 + 非機能要件）をAIに食わせると、設計パターンを数パターン提示してくれる。

```
入力: 「決済機能を作る。月間100万トランザクション。PCI DSS準拠」

AI出力:
  パターンA: マイクロサービス分割（スケーラビリティ○、複雑度△）
  パターンB: モジュラーモノリス（シンプル○、スケール△）
  パターンC: イベント駆動（疎結合○、デバッグ△）

→ 人間が選ぶ（Human-in-the-Loop）
→ 選んだパターンに基づいて設計書・コード雛形・テスト雛形が生成される
```

AIが勝手に決めない。候補を出して、人間が選ぶ。

### 柱2: 依存関係を管理して、変化点を伝播させる（★心臓部）

V字モデルの全ノード（要件・設計・コード・設定・DB・テスト）の依存関係をグラフで管理する。どこかを変更したら、そのグラフを辿って「ここも影響あるよ」を自動で出す。

```
stripe_adapter.pyを変更した
  → processor.pyに影響（コードの依存）
  → paypal_adapter.pyも直すべき（慣習の依存）
  → payment_feesマスターの整合性確認（データの依存）
  → 設計書も更新（ドキュメントの依存）
  → テストケース追加（テストの依存）
```

これを人間が頭で追うんじゃなく、依存グラフから自動で出す。

### 柱3: コンテキスト管理（AIに何を読ませるか）

AIに作業させるとき、「何をコンテキストに入れるか」が超重要。

- 全ソースを入れる → トークン数オーバー、または中間部分を忘れる（Lost in the Middle問題）
- 少なすぎ → 影響箇所を見落とす

CoDDは依存グラフから「今この変更に必要な情報だけ」を動的に選んでAIに渡す。しかも重要度に応じて全文/要約/メタ情報だけ、と解像度を変える。

### 3本柱の関係

```
柱1: 要件 → 設計候補を出す → 人間が選ぶ
柱2: 依存グラフで変化点を双方向に自動伝播  ★本丸
柱3: 依存グラフから必要な情報だけAIに渡す
```

柱2が本丸。柱1と柱3は柱2の依存グラフがあって初めて動く。

## 依存グラフとは何か

### 基礎

グラフ理論のグラフ。**ノード（点）** と **エッジ（線）** でできている。

```
[A] ---呼び出す--→ [B] ---読む--→ [C]
```

- ノード = モノ（ファイル、関数、設定キー、DBテーブル、設計書のセクション、等）
- エッジ = モノ同士の関係（AがBを呼ぶ、BがCを読む、等）

### 普通の依存グラフ（既存技術）

IDEの「参照を検索」がやっていることと同じ。ノードはソースコードの要素。エッジは `imports`, `calls`, `extends` みたいなコード上の関係。AST（抽象構文木）やLSP（Language Server Protocol）で自動検出できる。

### CoDDの依存グラフ：何が違うか

普通の依存グラフはコードの中だけ。CoDDはそこに**コード以外のモノを混ぜる**。

**ノードの種類が多い:**
```
コード:     processor.py, stripe_adapter.py, charge()
設定:       config.yaml の payment.provider キー
DBテーブル:  payment_fees マスター
設計書:     payment_flow.md のセクション3
テスト:     test_processor.py
バッチ:     nightly_settlement.py（夜間バッチ）
暗黙ルール:  「adapter間はインターフェースを揃える」という慣習
```

異種のモノが同じグラフに乗る。コードとDBと設計書とテストと暗黙知が、一つのグラフの中でエッジで繋がっている。

**エッジの種類も多い:**
```
imports      — ファイルをインポート（自動検出）
calls        — 関数を呼ぶ（自動検出）
reads_config — 設定値を読む（自動検出可能）
switched_by  — 設定値で呼び出し先が切り替わる（LLM推論 or FW解析）
driven_by    — マスターデータが挙動を制御する（LLM推論+人間注釈）
convention   — 慣習でセットで変えるべき（人間が教える or git履歴）
```

上の3つは機械的に取れる。下の3つはコードを読んだだけじゃ分からない。**ここがCoDDのフロンティア。**

### 具体例：決済処理モジュール

```
[processor.py::route_payment]
    │
    ├── reads_config ──→ [config.yaml::payment.provider]
    │                        │
    │                        ├── switched_by ──→ [stripe_adapter.py]
    │                        │                    （provider=stripeのとき）
    │                        │
    │                        └── switched_by ──→ [paypal_adapter.py]
    │                                            （provider=paypalのとき）
    │
    ├── driven_by ────→ [DB: payment_fees マスター]
    │                    （手数料率がロジックを制御）
    │
    └── implements ───→ [設計書: payment_flow.md セクション3]

[stripe_adapter.py]
    │
    └── convention ───→ [paypal_adapter.py]
                        （インターフェースを揃える慣習）
```

これが「熟練エンジニアの頭の中」の可視化。20年選手はこの構造を暗黙的に知っている。CoDDはこれを明示的なデータ構造にする。

## Q&A: 依存グラフのよくある疑問

### Q: データはどうやって保存するの？ YAMLファイル？ DB？

**段階で変わる。**

| 段階 | 保存形式 | 理由 |
|------|---------|------|
| 最初（小〜中規模） | YAML + SQLite | 手で書ける、gitで差分管理できる、軽い |
| 大規模になったら | SQLite（本体）+ Neo4j（検索用） | ノード数万・エッジ数十万になるとYAMLでは無理 |

**YAMLの例:**
```yaml
nodes:
  - id: src/payment/processor.py::route_payment
    type: method
    description: "決済ルーティング"

  - id: config/payment.yaml::provider
    type: config_key
    possible_values: [stripe, paypal, internal]

edges:
  - from: src/payment/processor.py::route_payment
    to: config/payment.yaml::provider
    type: reads_config
    confidence: 1.0
    source: auto_detected

  - from: config/payment.yaml::provider
    to: src/payment/stripe_adapter.py
    type: switched_by
    confidence: 0.95
    condition: "provider == 'stripe'"
    source: llm_inferred
```

YAMLのメリットは人間が読める・書ける・直せる・gitで差分管理できること。YAMLの限界はノードが1万個超えると検索が遅くなること。

中小規模のプロジェクトならYAMLだけで始めて、大きくなったらSQLiteに移行で十分。

使い分けのまとめ:
```
YAML ──→ 人間が書く注釈・慣習ルール・手動定義（入口）
SQLite ──→ グラフ本体。機械が高速に検索する場所（本体）
Neo4j ───→ パス探索・可視化。大規模時のみ追加（オプション）
JSON ────→ AIエージェントへの受け渡し（軽量スライス）
```

### Q: エッジに付いてる「信頼度」や「条件」って何？

CoDDのエッジはただの線じゃない。3つの重要な属性が付いている。

**属性1: 信頼度（confidence）— どのくらい信用できるか**

```
imports（コードがインポートしてる）     → 1.0（確実）
calls（関数を呼んでる）               → 1.0（確実）
switched_by（設定値で切り替わる）      → 0.95（ほぼ確実）
driven_by（マスターデータが制御）      → 0.8（結構確か）
convention（慣習で一緒に変える）       → 0.7（多分そう）
```

自動検出できるものは信頼度が高い。人間の知識やLLMの推論に頼るものほど低い。

**属性2: 証拠の出所（Evidence Ledger）— なぜこのエッジがあると言えるか**

同じエッジに複数の証拠が付くことがある。

```
stripe_adapter → paypal_adapter の慣習エッジ:
  証拠1: git履歴で12回中11回同時変更（history, score: 0.75）
  証拠2: PRレビューで「interfaceを揃えろ」コメント（human, score: 0.90）
  証拠3: LLMが「同じinterfaceだから」と推論（inferred, score: 0.60）
```

信頼度は証拠台帳から計算する。ベタ書きではなく、根拠から再計算可能にしておく。複数の弱い証拠が集まると信頼度が上がる仕組み（Noisy-OR）を使う。熟練者の「いくつかの匂いが一致したから怪しい」という直感のモデル化。

**属性3: 有効条件（condition predicate）— このエッジはいつ有効か**

```yaml
- from: config/payment.yaml::provider
  to: src/payment/stripe_adapter.py
  type: switched_by
  condition: "provider == 'stripe'"
```

本番環境では `provider=stripe` だからこのエッジは有効。でも開発環境では `provider=internal` かもしれない。そしたらこのエッジはその環境では無関係。

条件がないと全環境の依存が混ざって、「本番に関係ないのに影響ありって出る」ノイズが爆発する。熟練エンジニアは頭の中で条件を絞っている。それを明示的にデータとして持つ。

### Q: この依存グラフはどうやって作るの？ 手動で書くの？

**段階的に自動で作る。手動で全部書くのは不可能。**

#### Phase 1: 機械で取れるものを自動で取る（数分〜数十分）

コードをAST解析ツール（Tree-sitter等）で機械的に解析する。

```python
# このコードを解析すると:
from stripe_adapter import StripeAdapter

def route_payment(amount):
    provider = get_config('payment.provider')
    if provider == 'stripe':
        return StripeAdapter().charge(amount)
```

```
自動検出される:
  processor.py ── imports ──→ stripe_adapter.py （信頼度: 1.0）
  route_payment ── calls ──→ StripeAdapter.charge （信頼度: 1.0）
  route_payment ── reads_config ──→ payment.provider （信頼度: 1.0）
```

人間もLLMも不要。ASTパーサが構文を解析するだけ。**これだけで依存の60-70%は取れる。**

#### Phase 2: フレームワーク解析 + LLM推論（数時間）

Phase 1の `reads_config`（「設定を読んでる」）を起点に、**その先**を掘る。

**まずフレームワークに聞く（Spring Bootの場合）:**

Spring Boot Actuatorというフレームワーク標準の仕組みから、DIで何がどう繋がっているかが取れる。

```
GET /actuator/beans → 登録済みBean一覧と依存関係が全部出る
GET /actuator/conditions → @Profileや@ConditionalOnPropertyの評価結果が出る
```

LLMを使わずに、設定→実装の対応が取れる。最初の設計ではここを見落としていて「LLMに推論させるしかない」と書いていたが、実はフレームワークに聞けばかなり取れる。

**フレームワークで取れないものだけLLMに聞く:**

カスタムファクトリや独自のレジストリパターン等、機械的に解析できないものをLLMに読ませて候補を出す。ただしLLMがやるのは「依存を想像する」のではなく「パターンを分類して候補を提案する」こと。

**LLMは最後の手段。** 機械で取れる → フレームワークから取れる → それでも分からないところだけLLM。この順番。

#### Phase 3: 人間に確認する（継続的）

LLMの推論結果を熟練エンジニアに見せて、知識を引き出す。

```
CoDD:   「stripe_adapterとpaypal_adapterは
         config.payment.providerで切り替わると推論しました。
         合ってますか？」

熟練者: 「合ってる。あと、internal_adapterもあるよ。
         それとこの3つはインターフェースを必ず揃える必要がある」

→ ノード追加 + conventionエッジ追加
```

ゼロから「依存関係を全部書いてください」と言ったら誰も書かない。でも「これ合ってますか？他にありますか？」なら答えられる。LLMの推論を種にして、人間の暗黙知を引き出す。

#### Phase 4: 履歴から学習する（継続的・自動）

| データソース | 分かること | 生成されるエッジ |
|------------|----------|----------------|
| git log | 同時に変更されるファイルのペア | co_changed_with |
| 障害チケット | 「Xを変えたのにYを変え忘れて障害」 | incident_correlated_with |
| コードレビュー | 「ここも直さないと」コメント | convention_requires |
| テスト失敗ログ | Xの変更でYのテストが壊れた | validated_by_test |

時間が経つほど依存グラフが賢くなる。最初は弱いメンタルモデルでも、使い続けると熟練者に近づく。

### Q: SQLが動的に生成されてたらどうするの？

エンタープライズあるある。SQLが文字列結合やORMのメソッドチェーンで動的に組み立てられるケース。

```java
// 静的解析で追える
String sql = "SELECT fee_rate FROM payment_fees WHERE provider_code = ?";

// 静的解析で追えない
String sql = "SELECT " + columnName + " FROM " + tableName;

// もっと追えない
StringBuilder sql = new StringBuilder("SELECT fee_rate FROM payment_fees");
if (hasDiscount) {
    sql.append(" JOIN discount_master ON ...");
}
```

**3段構えで対処する:**

**第1段: 静的に取れるものは取る。** 素直なSQL文、ORMのアノテーション（`@Query`等）、HibernateやMyBatisのマッピング定義。これで6〜7割は拾える。

**第2段: 実際に動かして観測する（runtime observation）。** 動的SQLは実行されれば確定する。OTel（OpenTelemetry）でDBトレースを取ると、「実際にどのテーブルのどのカラムを読んだか」が記録される。推論より観測の方が確実。

```
trace記録の例:
  span: db-query
    sql: "SELECT fee_rate FROM payment_fees
          JOIN discount_master ON ...
          WHERE provider_code = 'stripe'"
    tables: [payment_fees, discount_master]  ← discount_masterが見つかった！
```

**第3段: LLM + 人間。** それでも取れない特殊ケース（年末だけ動くバッチ等）はLLMにパターン分析させ、熟練者に確認する。

現実の割合感:
```
静的解析で取れる:     60-70%
runtime観測で取れる:  15-20%
LLM推論で取れる:      5-10%
人間が教える:         5-10%
取れない:            数%（未観測のrare path）
```

100%は無理。でも90%取れれば実用上は十分。

### Q: runtime観測の結果はどうやって依存グラフに戻すの？

アプリにOTel（OpenTelemetry）を仕込んでおく。普通にアプリが動くとtraceが記録される。

```
本番/ステージング環境（普通に動いてる）
    │
    │  実行のたびにtraceが記録される
    │  「どの関数が、どのSQLを発行して、
    │    どのテーブルを読んで、その後何を呼んだか」
    │
    ▼
フィードバックパイプライン（夜間バッチ等）
    │
    │  traceデータを解析して依存グラフに反映:
    │  - 既知のエッジ → 証拠を追加、信頼度UP
    │  - 未知のエッジ → 新規作成（source: dynamic）
    │
    ▼
依存グラフが更新される
```

具体例: 静的解析では `payment_fees` テーブルしか検出されなかった。でもruntime観測で `discount_master` テーブルもJOINされていることが判明した。

```
改修前の依存グラフ（静的解析のみ）:
  StripeAdapter.charge → payment_fees     ← 知ってた

改修後の依存グラフ（runtime観測後）:
  StripeAdapter.charge → payment_fees     ← 知ってた
  StripeAdapter.charge → discount_master  ← ★新発見！
```

次に誰かがStripeAdapterを変更したとき、discount_masterも影響リストに出るようになる。

特別な作業は不要。アプリにOTelを入れて普通に動かすだけ。観測回数が増えるほど証拠が積み上がり、信頼度が上がる。

### Q: 改修前に全ソースのグラフを作っておくのか？ 新規開発では使えないのでは？

**新規開発と改修で使い方が違う。**

| フェーズ | CoDDの使い方 |
|---------|------------|
| 新規開発（ゼロイチ） | 柱1（設計生成）が主役。依存グラフはCIで裏で貯める |
| 成長期（機能追加が増える） | 依存グラフが効き始める。伝播エンジン投入 |
| 保守・改修 | **CoDDの本領発揮。** 全機能フル稼働 |

新規開発でアジャイルにぐるぐる回してるときは、毎日コードが激変するから依存グラフを見ても仕方ない。ただし**CIにPhase 1（静的解析）だけ仕込んでおく**。コミットのたびにグラフが自動更新される。使わないけど裏で貯まる。

v1.0が安定してきたら依存グラフが意味を持ち始める。運用・保守フェーズに入ったらフル活用。

**CoDDの本丸は「改修のための道具」。** 殿が最初に言った「エンタープライズの巨大システムでここ直したらどこが壊れるか」が核心だから、それは必然的に改修フェーズで最も効く。

### Q: シナリオごとに使い方が違うのでは？

その通り。4つのシナリオで使い方が変わる。

**シナリオ1: 新規開発**
- 柱1（設計生成）が主役。依存グラフはCIで裏で貯める
- Day 1に依存グラフを見ても仕方ない。将来への種まき

**シナリオ2: レガシーシステムの保守**
- CoDDが最も効く。「ここ直したらどこが壊れるか」の見落とし防止
- 最優先は熟練者が辞める前にPhase 3（知識吸い出し）を回すこと
- 全体に一気にやらない。変更頻度×障害頻度が高い領域から始める

**シナリオ3: モダナイゼーション（レガシー→モダン移行）**
- 依存グラフで「どこを先に切り離せるか」の移行計画が立てられる
- 旧→新の共有DB・共有APIのエッジを重点管理
- ストラングラーパターンとの相性が良い。旧ノードへのエッジがゼロ=安全に消せる

**シナリオ4: 既存システムのエンハンス（★最も多いケース）**
- 「新機能を入れること」自体は難しくない。「既存が壊れないことの保証」が難しい
- CoDDの価値は**設計前の影響範囲予測**。作ってから壊れるんじゃなく、設計段階で比較できる
- 設計案A vs 案Bの影響範囲（Amber帯の数）をグラフから定量比較して選べる

エンハンス特有の使い方:

| 使い方 | 保守 | エンハンス |
|--------|------|----------|
| 影響分析 | 変更後に「どこ壊れた？」 | 変更前に「どこ壊れそう？」 |
| 設計判断 | 不要（直すだけ） | 案A vs 案Bの影響範囲比較 |
| 回帰テスト | 壊れた箇所のテスト | 依存グラフから影響テストを自動抽出 |

回帰テスト計画にも効く:
```
従来: 「念のため全テスト回す」→ 3日
CoDD: 影響を受けるテストだけ自動抽出 → Amber帯50件+Gray帯30件
      影響なしの500件はスキップ可 → 3日が半日に
```

### Q: 育つ前（導入直後）のCoDDは役に立つのか？

Day 1で魔法は起きない。成長曲線はこう:

```
Day 1:   Phase 1のみ。grep + IDE検索と大差ない。ただしグラフとして構造化されてる
Week 1:  Phase 2も走らせる。設定駆動の依存が見え始める。grepを超え始める
Month 1: Phase 3で熟練者の知識が入る。慣習・暗黙知がグラフに乗る
Month 3: Phase 4が回り始める。「弱いメンタルモデル」として実用レベルに
```

育つ前でもCIに仕込んでおけば勝手に育つ。日常業務（コミット、レビュー、テスト、障害対応）が全部グラフを育てるデータになる。

一番のリスクは「熟練者が辞めてから慌てる」こと。育つ前でもPhase 3（人間に確認）だけは早めに回しておくべき。辞めてからでは聞けない。

## 変化点の伝播：変更したとき何が起きるか

### 基本の動き

```
1. 変更が入る（例: charge()のシグネチャ変更）
2. 変更点から、エッジを辿って隣のノードに行く
3. 隣のノードからさらに辿る
4. 辿るたびに「どのくらい影響ありそうか」のスコアが下がる
5. スコアが閾値を下回ったら止まる
6. 辿れた全ノードを「影響リスト」として出す
```

水面に石を投げたときの波紋のイメージ。変更点を中心に波が広がっていく。遠くに行くほど弱くなる。

### 「何を変えたか」で波の広がり方が変わる

同じファイルの変更でも、種類によって伝播パターンが全く違う。

| 変更の種類 | 波の広がり | 理由 |
|-----------|----------|------|
| publicメソッドのシグネチャ変更 | 広い | 呼び出し元が全部影響する |
| 内部実装だけの変更 | 狭い | 外から見た振る舞いは同じ |
| リファクタリング（意味変更なし） | ほぼ伝播しない | 名前変えただけ |
| 設定のデフォルト値変更 | 条件付きで広い | その設定を使ってる環境だけ |
| マスターデータの意味変更 | 最も危険 | ロジックの分岐が変わる |
| ドキュメントだけの修正 | 伝播しない | コードに影響なし |

変更の種類を見ずに一律で伝播させたら、ノイズだらけになって使い物にならない。

### 伝播の止め方

無限に波紋が広がったら全ノードに影響ありになって意味がない。止める基準が5つある。

```
① 契約境界で止める（最優先）
   public APIの契約自体が変わってなければ、外には伝播しない

② 条件が成立しない
   本番はstripeだけ → paypalの依存は本番環境では伝播しない

③ 新しい情報が増えない
   同じ種類の影響しか出ないなら止める

④ リスクスコアが低すぎる
   遠くに行くほどスコアが下がる

⑤ 深さ制限（最後の安全弁）
```

### 影響リストの見せ方：3帯域分類

影響リストを重要度で3つの帯に分ける。

**Green（自動更新していい）:**
```
1. [auto] test_processor.py — 引数変更に追従
2. [auto] docs/api/payment.md — APIシグネチャ更新
```
信頼度が極めて高い。機械的に正解が決まる。AIが直して、人間はdiffを確認するだけ。

**Amber（人間がレビュー必須）:**
```
3. [review] stripe_adapter.py — インターフェース整合性
4. [review] paypal_adapter.py — 同上（★本番paypal環境のみ影響）
5. [review] DB: payment_fees — 手数料率マスターとの整合性
```
見落としたら障害になる箇所。多少ノイズがあっても漏らさないことを優先。

**Gray（参考情報）:**
```
6. [info] nightly_settlement.py — 夜間バッチへの影響
7. [info] monthly_report.py — 月次レポート
```
関係あるかもしれないレベル。余裕があったら見る。

成果物の種類によって閾値が変わる。料金計算コードなら少しでも怪しければAmber。READMEなら余程のことがなければGray。

## Q&A: 伝播エンジンのよくある疑問

### Q: 自動的に全部直してくれるの？ それとも「ここ見ろ」だけ？

**帯によって変わる。**

| 帯 | AIの役割 | 人間の役割 |
|----|---------|----------|
| Green | **AIが直す** | diffを確認してOK押すだけ |
| Amber | **AIが修正案を複数出す** | どれを採用するか選ぶ |
| Gray | **AIが確認ポイントを教える** | 自分で読んで判断する |

Amber帯の例:
```
CoDD: 「charge()のシグネチャが変わりました。
       paypal_adapter.pyにも影響があります。

       修正案A: 同じシグネチャに変更
       修正案B: processor.pyのルーティング側で吸収

       → どちらを採用しますか？ [A/B/手動で別案]」
```

全部人力じゃない。全部自動でもない。帯によってAIと人間の役割分担が変わる。

「全自動で直す」は最終ゴール。最初は「見落とし防止のセーフティネット」として出す。信頼が積み上がったらGreen帯（自動更新の範囲）を広げていく。

### Q: Green帯は最初少ないけど、増えていくの？ なぜ？

**チームが普通に仕事するだけで、証拠が貯まって信頼度が上がる。**

最初（LLM推論だけ）:
```
stripe_adapter → paypal_adapter の慣習エッジ:
  証拠1: LLMが推論（score: 0.60）
  → 信頼度: 0.60 → Amber（人間がレビュー）
```

3ヶ月後（証拠が積み上がった）:
```
stripe_adapter → paypal_adapter の慣習エッジ:
  証拠1: LLMが推論（score: 0.60）
  証拠2: git履歴で12回中11回同時変更（score: 0.75）
  証拠3: レビューで「interface揃えろ」コメント（score: 0.90）
  証拠4: 片方だけ変えたら結合テスト落ちた（score: 0.95）
  → 信頼度: 0.9995 → Green（自動更新OK）
```

コミットする、レビューする、テストが落ちる、障害が起きる。全部がエッジに証拠を追加する。特別な作業をしなくても日常業務がグラフを育てる。

逆にコードを大きくリファクタしたら既存の証拠が古くなり、信頼度が下がって自動的にGreen→Amberに戻る。自浄作用がある。

### Q: 個人の知識じゃなく、組織で育てた方がよくない？

**その通り。それがCoDDの本質的な価値の一つ。**

今の現実:
```
田中さん（20年選手）の頭の中: 「ここ変えたらあそこも」
佐藤さん（15年選手）の頭の中: 「あのマスターはあのバッチに影響」
鈴木さん（10年選手）の頭の中: 「この設定は本番だけ有効」
→ 全部バラバラに各人の頭の中。退職したら消える。
```

CoDDの依存グラフに入れると:
```
田中さんが教えた → conventionエッジ（evidence: human:tanaka）
佐藤さんが教えた → driven_byエッジ（evidence: human:sato）
鈴木さんが教えた → condition: "env == 'prod'"（evidence: human:suzuki）
→ 個人の暗黙知が、組織の共有資産（グラフ）に変換される。
→ 誰が教えたか記録される。他の人が修正できる。人が辞めても残る。
```

Phase 3（人間に確認）とPhase 4（履歴から学習）が組織学習そのもの。レビューのたびに、コミットのたびに、障害対応のたびに、チーム全員の知識がグラフに蓄積される。

| 問題 | CoDDでの解決 |
|------|------------|
| 属人化 | 暗黙知がグラフに記録される。人が辞めても残る |
| 知識の断絶 | 新メンバーはグラフを見ればキャッチアップできる |
| オンボーディング | 「ここ変えたらどこ見ろ」がシステムから出る |
| レビュー品質のばらつき | グラフがレビュー観点を補完する |

## 柱3: コンテキスト管理 — AIに何を読ませるか

### 問題

影響リストが出た。Amber帯のやつをAIに修正案を出させたい。でもAIに何を読ませるかで修正の質が激変する。

```
悪い例1: ソース全部入れる（10万行）
  → トークン数オーバー。そもそも入りきらない。
  → 入ったとしても、中間部分を忘れる（Lost in the Middle問題）

悪い例2: 変更対象のファイルだけ入れる
  → 「このメソッドは何のためにあるのか」が分からない
  → 設定値やマスターデータとの関係が見えない
  → 的外れな修正案が出る
```

### Lost in the Middle問題

LLM（大規模言語モデル）のアテンション機構の弱点。

```
コンテキストウィンドウ:
┌──────────────────────────────────────┐
│ [先頭] ← よく覚えてる                  │
│                                      │
│ [中間] ← ここを忘れがち ⚠️             │
│                                      │
│ [末尾] ← よく覚えてる                  │
└──────────────────────────────────────┘
```

先頭と末尾はよく覚えてるけど、中間に置いた情報は忘れやすい。だから重要なコードを中間に置くと、AIがそれを無視して修正案を出す。

### CoDDの解決策: 依存グラフがコンテキストを自動構成する

依存グラフがあるから、「今この変更に何が関係するか」が分かる。それを使ってAIに渡す情報を自動で選んで、最適な順番で配置する。

```
コンテキストウィンドウの構成:

┌──────────────────────────────────────┐
│ [先頭: よく覚えてる]                    │
│                                      │
│  1. 変更対象ファイル（全文）             │
│  2. 変更の意図・目的                    │
│  3. 直接関係するインターフェース（全文）   │
│                                      │
├──────────────────────────────────────┤
│ [中間: 忘れやすい → だからメタ情報だけ]    │
│                                      │
│  4. 間接的に関係するモジュール（要約のみ） │
│     「このモジュールは○○する」程度        │
│  5. 設定値の一覧と現在値                │
│  6. マスターデータの構造（DDLのみ）       │
│                                      │
│  ※ コード全文は入れない。忘れても         │
│    致命的じゃないレベルの情報だけ          │
│                                      │
├──────────────────────────────────────┤
│ [末尾: よく覚えてる]                    │
│                                      │
│  7. 影響セットのサマリー表               │
│  8. 過去の関連障害情報                   │
│  9. チェックリスト                      │
│     「以下を確認してください」            │
│                                      │
└──────────────────────────────────────┘
```

3つのキモ:

**キモ1: 先頭に変更対象、末尾にチェックリスト。** アテンションが高い位置に重要な情報を置く。

**キモ2: 中間にはコード全文を入れない。** メタ情報（「このモジュールは決済処理をする」程度の要約）だけ。忘れても致命的じゃない。

**キモ3: 必要になったらその場で読み込む。** 中間のメタ情報を見てAIが「ここをもっと詳しく見る必要がある」と判断したら、その時点でそのファイルを全文読み込む。最初から全部入れない。

### 「何を入れるか」の判断基準

重要かどうかじゃなく**曖昧かどうか**で決める。

```
重要だけど曖昧じゃない → 要約で十分
  例: 「このモジュールはStripe APIを呼ぶ」（分かりきってる）

重要で曖昧 → 全文を読み込む
  例: 「このモジュールの分岐条件がどうなっているか不明」

重要じゃない → メタ情報だけ or 入れない
```

多くの失敗は「重要だから全文を入れよう」と考えることから起きる。本当は「曖昧だから全文が要る」。重要でも分かりきってることは要約で十分。

### 依存グラフとの連携

```
1. 伝播エンジンが影響リストを出す
2. 各ノードについて「全文/抜粋/要約/メタ情報」のどれを入れるか判定
3. トークン予算内で最大限有用な組み合わせを選ぶ
4. U字型に配置（先頭と末尾に重要なもの）
5. AIに渡す
6. AIが「ここの全文が必要」と判断したら追加読み込み
```

依存グラフがないと、何を入れるか人間が毎回判断しないといけない。依存グラフがあるから「この変更にはこの情報が必要」を自動で決められる。これが柱2と柱3の連携。

## 3本柱はループしている

3本柱は独立した機能じゃない。全部繋がって、循環している。

### 全体フロー

```
【柱1】要件が来た
  │
  │  「決済機能に多通貨対応を追加」
  │
  ▼
  AI が依存グラフに対する「変更案」を複数生成
  │
  │  案A: currencyカラム追加 + adapter修正 + マスター追加
  │  案B: 通貨変換レイヤーを新設 + 既存は触らない
  │
  ▼
  人間が選ぶ → 案Aを採用
  │
  │  → この時点で「何が変わるか」がグラフ上で確定
  │
  ▼
【柱2】依存グラフで影響を伝播
  │
  │  案Aを選んだことで:
  │  - payment_feesにcurrencyカラム追加（SchemaChanged）
  │  - processor.pyの引数変更（InterfaceContractChanged）
  │  - stripe/paypal両adapterに影響（switched_by経由）
  │  - 設計書のセクション3に影響（implements経由）
  │  - テスト仕様に多通貨テスト追加（validated_by経由）
  │  - 夜間バッチに影響の可能性（co_changed_with経由）
  │
  │  → Green / Amber / Gray に分類
  │
  ▼
【柱3】AIに最適なコンテキストを渡す
  │
  │  Amber帯のstripe_adapter修正のために:
  │  - 先頭: stripe_adapter.py全文 + processor.pyのインターフェース
  │  - 中間: paypal_adapterの要約 + payment_feesのDDL
  │  - 末尾: 「currencyの扱いを統一すること」チェックリスト
  │
  ▼
  AIが修正案を出す
  │
  │  「stripe_adapter.pyをこう直したらどうですか？（案A/案B）」
  │
  ▼
  人間がレビュー → 採用 → コミット
  │
  ▼
【柱2に戻る】コミットで依存グラフが自動更新
  │
  │  Phase 1が走る → 新しいエッジが追加/更新
  │  証拠が1つ増える → 信頼度が少し上がる
  │
  ▼
  次の変更で、さらに精度が上がった状態で伝播する
```

### ループ図

```
柱1（設計生成）──→ 柱2（影響伝播）──→ 柱3（コンテキスト最適化）
      ↑                                        │
      │                                        │
      └──────── フィードバック ──────────────────┘
                （修正結果がグラフを育てる）
```

- 柱1が「何を変えるか」を決める
- 柱2が「どこに影響するか」を出す
- 柱3が「AIに何を読ませるか」を決める
- AIが修正案を出す → 人間が確認 → コミット
- コミットが柱2のグラフを育てる
- 次のサイクルで精度が上がる

**回せば回すほど賢くなる。3本柱はバラバラの機能じゃなく、1つの循環システム。**

## CoDDの本質: 組織の知識と経験の集合体

技術的には「変更影響分析ツール」。でも本質は違う。

```
依存グラフ = ソースコードの構造（機械が読める部分）
           + フレームワークの設定情報（機械が聞ける部分）
           + 実行時の挙動（機械が観測できる部分）
           + 熟練者の経験知（人間から引き出した部分）
           + チームの歴史（git/障害/レビューから学んだ部分）
```

つまり**「そのシステムに関わった全員の知識と経験」がグラフに結晶化する。**

### なぜ従来のナレッジマネジメントは失敗するのか

| 既存のアプローチ | 問題 |
|---------------|------|
| ドキュメント化（Confluence/Notion） | 書かない。書いても更新されない。腐る |
| 引き継ぎ資料 | 辞める直前に慌てて書く。漏れだらけ |
| ペアプロ/モブプロ | 知識は移転するが記録されない |
| コードコメント | 書かない。書いても「何」であって「なぜ」がない |

### CoDDの依存グラフが違う理由

CoDDの依存グラフは「わざわざ書く」のではなく「日常業務の副産物として勝手に育つ」。

- CIが毎コミット自動更新（腐らない）
- runtimeが毎日観測（自動で育つ）
- レビューのたびに人間の知識が入る（使うついでに育つ）
- 障害が起きたら自動で証拠が追加される（痛い目が知識になる）
- 「なぜ」が残る — Evidence（証拠）に「前回の障害」「レビューコメント」が記録される
- 人が辞めても残る — グラフに結晶化されている

### CoDDを一言で表すと

```
技術的な説明:
  「条件付き証拠グラフによる変更影響分析」

人間に刺さる説明:
  「チーム全員の知識と経験が、使うほど育つ依存グラフに結晶化する。
   人が辞めても、知識は残る。」
```

## CoDDの進化の先: AI自動修正・半自動エンハンスへ

依存グラフが育つほど、AIによる自動化の範囲が広がっていく。

### Level 1: 影響を教える（最初の段階）
```
人間: 「ここ変える」
CoDD: 「ここも影響あるよ」（Green/Amber/Gray）
人間: 自分で直す or AIの修正案を選ぶ
```

### Level 2: 修正をほぼ自動でやる（Green帯の拡大）
```
人間: 「ここ変える」
CoDD: 影響を特定 + Green帯を自動修正 + Amber帯の修正案を提示
人間: Amber帯だけ判断する
```

### Level 3: 障害を自動修正する
```
監視: 「本番でエラー発生」
CoDD: エラー箇所 → 依存グラフで原因候補を特定
      → 「直近のcommit Xが原因の可能性95%」
      → 修正パッチを自動生成 → テスト自動実行 → パス
人間: パッチをレビューしてマージするだけ
```
依存グラフに「このcommitでこのエッジが変わった」が記録されてるから、エラー箇所から逆に辿れば最近変わったエッジにたどり着く。

### Level 4: エンハンスを半自動でやる
```
PM: 「サブスクリプション機能を追加して」
CoDD: 要件を解析 → 変更案を複数生成 → 影響範囲を比較
      → 最もリスクが低い案を推奨
      → コード・テスト・設計書・migrationを生成
人間: 設計判断 + 最終レビューだけ
```

### なぜCoDDがあるとこれができるか

普通のAIツール（Cursor、Copilot等）との違い:

| | 普通のAI | CoDD搭載のAI |
|--|---------|-------------|
| コンテキスト | 今開いてるファイル+周辺 | 依存グラフから動的に構成 |
| システム全体の影響 | 分からない | 分かる |
| 設定駆動の依存 | 見えない | 見える |
| 過去の障害パターン | 知らない | 知ってる |
| 熟練者の知識 | ない | グラフに結晶化されてる |

AIの能力は同じ。与えるコンテキストの質が桁違い。

### 到達条件

Level 1-2はすぐできる。Level 3-4に必要な条件:
- 十分に育った依存グラフ（Month 3+）
- テストカバレッジが高い
- CI/CDパイプラインが整備されてる
- Green帯のprecisionが95%以上

Level 1-2をやってるだけで自動的にLevel 3-4に向かって進む。日常業務がグラフを育てて、精度が上がって、自動化の範囲が広がる。

## CoDDの限界（正直に）

### 原理的に解けないこと

1. **記録されていない意図は復元できない。** コードにもドキュメントにもtraceにも残っていないビジネス意図は弱い
2. **まれな経路は観測しにくい。** 年に1回しか動かないバッチは観測データが集まらない
3. **履歴データは悪い慣習も学習する。** 過去にバグ込みでco-changeしていたパターンも覚える
4. **100%は無理。** 「未知の依存がある可能性」は常に残る

### 実務的にはどの程度問題か？（Q&Aから）

> **Q: 限界1「意図が消える」って言うけど、コード解析したら構造は残ってるから依存は追えるのでは？**

その通り。「なぜそうなっているかの理由」は消えても、「ここを変えたらここに影響する」という構造的依存はAST解析で見える。影響分析には十分使える。限界1は「意図の説明ができない」であって「依存が見えない」ではない。

> **Q: 稀な経路が観測できないって言うけど、稀な経路はそもそも触らなくない？**

その通り。年1回のバッチを日常的に改修することはまずない。例外はモダナイゼーション（システム全体リプレース）のときだけ。そのときは意図的にその経路を1回動かしてトレースを取ればいい。

> **Q: 結局これらの限界って致命的なの？**

致命的ではない。どれも「だからCoDDが使えない」にはならない。このセクションは**論文のお作法**の側面が強い。学術論文では「うちの手法は万能です」と書くと査読で落とされるため、自分から弱点を明示しておく。「原理的にここはできません、でも実務上は問題になりにくいです」と書くことで、逆に信頼度が上がる。

### 限界3「悪い慣習の学習」への処方箋

> **Q: 履歴データが間違った依存を覚えちゃうなら、何か対策はないの？**

3つの処方箋がある。

**処方箋1: バグ修正コミットとの突合。** AとBのco-changeの後に、Bを元に戻すrevertやバグ修正commitが頻繁にあるなら、「A→Bのco-changeはバグだった可能性が高い」と判定して信頼度を下げる。

**処方箋2: Evidence Ledgerの多数決。** 履歴ベースの証拠が「A→B」と言っていても、静的解析・フレームワーク解析・実行トレースのどれもA→Bを裏付けてなければ、Noisy-ORで計算しても信頼度は低いまま。履歴だけが言ってる依存は自動的にAmber（要確認）に分類される。

**処方箋3: 人間フィードバックの上書き。** Phase 3で人間が「この依存は間違い」とマークしたら、Evidence Ledgerにnegative evidence（否定証拠）として記録される。以降、同じパターンが出てきても信頼度が下がる。

完全には防げないが、「悪い学習が暴走する」ことは防げる。**履歴だけの証拠が単独でGreen帯に入ることはない設計**になっている。

### だからCoDDは完璧を目指さない

目標は「熟練者の7割のカバレッジ」。残り3割は:
- テスト（変更後に壊れたら気づく）
- 監視（本番で異常があれば気づく）
- 人間のレビュー（最後の砦）

で補う。CoDDは**セーフティネット**であって、銀の弾丸ではない。

## CoDDは「AI中心」ではなく「AIを活かすためのインフラ」

> **Q: CoDDって完全にAIセントリックな開発手法じゃない？**

一見そう見えるが、違う。

柱1（依存グラフ）と柱2（伝播エンジン）は**完全にdeterministic**。AST解析、静的解析、フレームワーク解析、SQLパーサ。AIは使っていない。ここが土台。

CoDDの構造:
- **土台**: 決定論的な依存グラフ + 伝播アルゴリズム（AIゼロ）
- **中間層**: AIが「足りない部分」を補完（Phase 2のLLM分類、Layer 3-5の推論）
- **出力層**: AIが「結果を使って仕事する」（コンテキスト最適化、自動修正）

AIが中心にいるのではなく、**AIが仕事できるようにデータを整えてあげるインフラ**。

### 既存AI開発ツールとの違い

今のClaude CodeやCopilotは毎回ゼロからコードベースを読んで依存関係を推測している。「毎朝記憶喪失になるベテランSE」のようなもの。

| 項目 | CoDDなし | CoDDあり |
|------|----------|----------|
| 依存の把握 | AIが毎回推測 | 事前計算済みのグラフを参照 |
| 影響範囲 | AIが「たぶんここも」 | 証拠付きで「ここ、理由はこれ」 |
| コンテキスト | 全部突っ込むか勘で選ぶ | 必要なものだけ最適配置 |
| 修正提案 | コード片だけ見て提案 | 上流〜下流を見て提案 |

**結論**: CoDDは「AIセントリックな開発手法」ではなく、**「従来の開発をAI-readyにするインフラ」**。AIを中心に据えるのではなく、AIが最大限に力を発揮できるようにシステムの知識を構造化する。主役は「構造化された依存知識」で、AIはそれを使う消費者。

既存AI開発ツール（Copilot、Cursor、Claude Code）は「AIを賢くする」方向。CoDDは「AIに食わせるデータを賢くする」方向。これが決定的な違い。

### 主体の逆転: 人間がAIをサポートする時代へ

> **Q: これって「AIが人間をサポート」じゃなくて「人間がAIをサポート」に逆転してない？**

その通り。従来のAI開発支援は「人間が設計→人間がコーディング→AIが補完」。主体は人間で、AIはアシスタント。

CoDDがLevel 3-4に進化すると「AIが影響範囲を特定→AIが修正案を生成→人間が承認する」になる。**主体がAIで、人間は「合ってるか確認する」レビュアー**になる。

これはparadigm shift（パラダイムシフト）。GPT-5.4がCoDDを「vision paper向き」と評価したのは、まさにこういう大きな方向性の転換を論じるのに適しているから。

### 「AI駆動組織」への完璧な基盤

2025-2026年の世の中の流れは「AI駆動組織」「AI-native企業」。AIが主体で人間が補完する働き方。でも「ソフトウェア開発で具体的にどう実現するか」の方法論は誰も答えていない。

今のAI開発ツール（Copilot、Cursor、Claude Code、Devin）は全部「人間→AI」方向。AIが主体的に判断するための知識基盤がない。だから毎回人間が依存関係を教えないといけない。

CoDDはここに答えを出す。AIが自律的に開発判断するための構造化された知識基盤を提供する。

- もっと賢いAI → モデル進化待ち（CoDDの範囲外）
- もっと良いプロンプト → 人間依存のまま（スケールしない）
- **AIが参照できる構造化された知識** → CoDDが提供するもの

**CoDDは「AI駆動開発」を実現するための具体的な方法論**。技術の新規性よりも、「問いの立て方自体が新しい」ことが価値。

vision paperとしての核心: "While current AI-assisted development tools augment human developers, the emerging paradigm of AI-driven organizations demands the inverse: structured knowledge infrastructure that enables AI to lead development decisions, with humans providing oversight."

### ソフトウェア開発を超えた汎用性

> **Q: CoDDって開発だけじゃなくて、組織活動全般に応用できるのでは？**

できる。CoDDが解いてる問題を抽象化すると「複雑なシステムで、ある要素を変えたとき、他のどこに影響が出るか」。これはコードに限った話ではない。

組織で同じ問題が起きている例:
- **人事異動**: Aさんが抜けると、Bプロジェクトの暗黙の窓口が消える → 引き継ぎ漏れ
- **社内規程変更**: 経費規程を変えたら、承認フローと会計システムの設定も変える必要がある
- **法改正対応**: インボイス制度 → 請求書、会計仕訳、取引先マスタ、契約書雛形、全部影響
- **組織改編**: 部署統合 → 権限テーブル、メーリングリスト、稟議ルート、座席表、全部変わる

CoDDの3本柱をそのまま置き換え可能:
- コードの依存グラフ → 業務プロセス・規程・人・システムの依存グラフ
- 条件付きエッジ → 「この部署だけ」「この契約形態だけ」
- Evidence Ledger → 根拠（規程、契約書、議事録）
- 伝播エンジン → 「ここ変えたらここも変えろ」の自動通知
- コンテキスト最適化 → 担当者に関係ある変更だけ届ける

ただし論文戦略としてはまずソフトウェア開発に絞るのが正解。「なんにでも使えます」は査読で「何も証明してない」と言われる。cs.SEで実績を作ってから汎用化する。

## 2つのレイヤー: コード内の伝播 と 開発工程間の伝播

CoDDの依存グラフには2つのレイヤーがある。

### レイヤー1: コード内の依存伝播
`Service.java → Repository.java → テスト` のような、システム内部の依存。柱2の伝播エンジンが担当。

### レイヤー2: 開発工程の成果物間の伝播
`要件定義書 → 基本設計 → 詳細設計 → コード → テスト仕様書 → 運用手順書` のような、V字モデルの成果物間の依存。柱1の「設計書⇔コードの双方向同期」が担当。

### 1つのグラフに全部入っている

CEGのノードタイプは成果物もコードも含む:
- requirement（要件定義の項目）
- design（基本設計・詳細設計のセクション）
- code（クラス、メソッド、設定ファイル）
- test（テストケース、テスト仕様書の項目）
- operation（運用手順書、監視設定）
- config（application.yml、環境変数）
- data（テーブル定義、マスターデータ）

レイヤー1（コード内）もレイヤー2（工程間）も**同じグラフ、同じ伝播エンジン**で扱う。

### 双方向がCoDDの真骨頂

上→下だけでなく、下→上にも伝播する。`point_master`にカラム追加 → 詳細設計にカラム定義追記 → 基本設計のER図更新 → 要件定義の確認。今までこれを全部人間が頭で追っていたのを、グラフを辿って自動で「ここも見ろ」を出す。

## まとめ

```
CoDD = 条件付き証拠グラフ（依存の管理）
     + 変化点伝播エンジン（影響の自動特定）
     + コンテキスト最適化（AIへの情報選択）
```

**提供するもの:**
- 「何が関係ありそうか」（影響リスト）
- 「どこまで見れば安全か」（Green/Amber/Gray）
- 「なぜそう言えるか」（証拠 + 影響パスの説明）
- 「組織の知識の蓄積」（退職しても残るナレッジベース）

これが実務で使える"弱いメンタルモデル"。
