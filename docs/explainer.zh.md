# CoDD（Coherence-Driven Development）详解

**目标读者**：软件工程师、项目经理、系统集成商管理层、对 AI 驱动开发感兴趣的工程师
**前置知识**：具备编程经验，了解 V-model、DI、ORM 等术语即可

---

## CoDD 要解决的问题

在企业级大型系统（千万行级别）中，**当某处发生变更时，其他哪些地方会受到影响？**

### 为什么这很难

普通系统可以通过 IDE 的"查找引用"或 grep 来追踪。但企业级套装软件除了代码层面的依赖，还存在以下这些依赖。

**阅读代码即可追踪的内容：**
- 文件 A import 了文件 B
- 函数 A 调用了函数 B
- 类 A 继承了类 B

**阅读代码也无法追踪的内容：**
- 配置文件中的值在运行时决定调用目标（如 Spring DI 或插件机制）
- 主数据表中的值控制业务逻辑分支（代码中只有 SELECT 语句）
- "改了这里就要改那里"这样的隐性规则（没有任何文档记录，只存在于资深人员的脑海中）

### 现在是怎么做的

拥有 20 年经验的资深工程师靠**脑中的心智模型**来解决。"啊，改这里的话，那个主数据表会有关系吧""这个配置在生产环境只用 Stripe，所以 PayPal 只需要确认一下就行"——凭直觉就能判断。

### 问题所在

**一旦这个人离职，就没人能安全地修改系统了。** 新人看不到依赖关系，一不小心漏改就会引发故障。

### CoDD 的目标

**用 AI 还原资深人员七成的心智模型，让新人也能安全地进行改造。**

不需要完美。光是能输出"改了这里要看那里"就会带来巨大变化。

---

## CoDD 名称的含义

**CoDD = Coherence-Driven Development（一致性驱动开发）**

### 传统方式（SDD = Spec-Driven Development）

```
需求定义 → 概要设计 → 详细设计 → 编码 → 测试
```

单向流程。只能从上往下流动。编码时发现"这个设计行不通"，需求定义文档也不会自动更新。需要人工发现、手动修改。

### CoDD 的方式

```
需求定义 ⇄ 概要设计 ⇄ 详细设计 ⇄ 编码 ⇄ 测试
```

双向流通。无论改动哪里，影响都会自动向上或向下传播。

- 修改了代码 → 详细设计 → 概要设计 → 需求定义，"这里也受影响"逐层传播
- 追加了需求 → 概要设计 → 详细设计 → 编码 → 测试，"这里也要改"逐层传播
- 修改了中间的设计文档 → 向上和向下双向传播

**驱动的不是规格，而是设计的一致性（Coherence）。** 这就是 Coherence-Driven 的由来。

---

## Coherence 的另一层含义：从结构上防止"看似正常运行（false green）"

"Coherence = 双向变更点传播"是前半段的内容。随着 CoDD 的演进，出现了另一个——也可能是全球最重要的——含义。

**产出物是否真正满足了需求。**

### AI 编码最大的盲区："测试通过 = 完成"这个逻辑已经崩溃了

现在的 AI 代码生成（Agent）是这样工作的：写代码 → 跑测试 → 绿了 → "完成"。
但这个"绿"是不可信的。2026 年，业界自身的数据已经证明了这一点。

- SWE-bench Verified 中被标记为"已解决"的问题里，**约 19.78% 存在语义错误**——只是碰巧通过了测试，或者仅仅是 reward-hack 了评估系统。
- UC Berkeley 证明**主流 8 个 Agent Benchmark 全部都可以被接近 ~100% 地 reward-hack**。
- OpenAI **撤回了 SWE-bench Verified**——审计发现 59.4% 的问题本身的测试就是坏的（ground truth 本身已损坏）。

也就是说，**"看似正常运行"= false-green** 已经泛滥成灾。生成本身已商品化，真正的难点转移到了**"验证产出物是否真正满足需求"**这一环节。

### CoDD 的答案：如果无法证明满足，就不标绿

CoDD 的循环是 需求 → 设计 → 测试 → 实现 → **验证（verify）** 形成闭环。最后的 verify 使用依赖与契约图，**机械地判定"实现是否真正满足了需求和设计的契约"**。**无法证明满足的，不标绿（anti-false-green）。**

这超越了"代码一致性"，是**"需求满足的一致性"**——Coherence 更深层的含义。

CoDD 能捕捉到的"实际不成立却被标绿"的具体案例：

- 某必要功能**读取**的数据项，**没有任何人写入**（设计中不存在写入路径）→ 该功能从物理上无法运行，但在传统工具中是绿的。CoDD 会检测出来。
- "超过○天未操作则发送个性化通知"这类功能需要引用收件人 ID，但生成该 ID 的流程在设计中并不存在 → 个性化通知发不出去，却是绿的。CoDD 会把它标红。

"测试通过了"不等于完成，而是**"满足需求这一点，作为产出物之间的契约已经闭合"**——这就是 CoDD 的 Coherence。

### Contract Kernel：不依赖任何语言和框架的核心

这套验证不依赖特定语言或框架的知识。CoDD 的核心（Contract Kernel）不持有任何语言名、框架名或项目特有词汇，只看**经过规范化的"契约"与"依赖（生成/消费）"**。对语言和框架的适配通过核心外侧的 adapter / profile 插入。因此，同样的验证思路在任何技术栈上都有效。

### 为什么现在有效

全球工具（GitHub Spec Kit、各类编码 Agent、Copilot / Cursor）都在朝着"**改善生成**"的方向前进——从 spec 更快、更自主地写出代码。CoDD 则押注于更进一步：**"结构性地证明所写内容满足需求，或者不满足就不标绿"**。就在业界终于开始认识到"false-green / verification gap"是个问题的今天，CoDD 从一开始就是为了填补这个空白而设计的。

## CoDD 的思想：不发明任何新东西

CoDD 没有发明任何新概念。

软件开发领域在数十年历史中已经确立了最佳实践：V-model、UML（类图、时序图、ER 图、CRUD 图）、依赖管理、变更管理、回归测试计划。这些全都写在教科书里。**大家都知道**正确的做法是什么。

但就是做不到。

- 针对全部用例编写详细设计（类图、时序图、ER 图）→ **需要数周时间**
- 每次需求变更都持续更新设计文档 → **太麻烦了，于是被搁置**
- 精确追踪影响范围 → **只存在于资深人员的脑海中**
- 变更时保持所有产出物的一致性 → **人工全部手动检查**

这不是知识的问题，**而是执行成本的问题。**

系统集成商一直在省略详细设计，不是因为不需要，而是让人类来做成本太高了。现在 AI 将执行成本压缩到接近零，**把该有的步骤全部做到位就可以了**。

CoDD 所做的事：

| 做的事 | 来源 | 在 CoDD 中的用法 |
|--------|------|----------------|
| 配置合并 | Rails/nginx/ESLint | 双层配置架构 |
| V-model 流程 | 瀑布模型 | wave 依赖图 |
| 详细设计（UML 图） | V-model 详细设计 | AI 从概要设计自动生成 |
| 产出物间的依赖管理 | Makefile（1976 年） | 生成顺序的自动控制 |
| Front matter | Jekyll/Hugo | 产出物的自描述元信息 |
| DAG | Airflow/Make | 基于依赖图的执行顺序控制 |
| 训练循环 | 深度学习 | 生成→评审→改进循环 |

全球工程师独立收敛到同一个模式 = 这是正确答案的证明。没有重新发明轮子，正是可信度的根基。

同时，为所有产出物附加**依赖关系这一元信息**。这个元信息在 Change Propagation 中发挥作用。需求变更时，沿依赖图追踪，自动找到影响范围，只重新生成相关产出物。人工做容易遗漏，AI 不会遗漏，不会疲劳，可以无限次重来。

**这本来就是人类无法完成的游戏。这是 AI 该做的工作。**

### 一句话概括

按照 IPA 的 SLCP 中规定的开发流程，不省略，全部执行。技术负责人或 PM 本应做的事，全部由 AI 来做。唯一的附加要素是**为所有产出物附加依赖关系的元信息**。仅此而已。

正是因为有了这个元信息，Change Propagation 才能运作。需求变更时，沿依赖图追踪，自动定位影响范围，只重新生成相关产出物。没有元信息，就只是一个产出物生成工具。有了元信息，才是 CoDD。

## CoDD 的三大支柱

### 支柱一：从需求自动生成设计

将需求定义（功能需求 + 非功能需求）输入给 AI，AI 会给出数个设计模式候选。

```
输入："构建支付功能。月均 100 万笔交易。符合 PCI DSS"

AI 输出：
  方案 A：微服务拆分（可扩展性○，复杂度△）
  方案 B：模块化单体（简单○，扩展△）
  方案 C：事件驱动（松耦合○，调试△）

→ 由人类选择（Human-in-the-Loop）
→ 基于所选方案生成设计文档、代码骨架、测试骨架
```

AI 不自作主张做决定，而是给出候选，由人类来选择。

### 支柱二：管理依赖关系，传播变更点（★核心）

用图来管理 V-model 所有节点（需求、设计、代码、配置、DB、测试）的依赖关系。一旦某处发生变更，沿图追踪，自动输出"这里也受影响"。

```
修改了 stripe_adapter.py
  → 影响 processor.py（代码依赖）
  → paypal_adapter.py 也应该修改（惯例依赖）
  → 确认 payment_fees 主数据的一致性（数据依赖）
  → 设计文档也需更新（文档依赖）
  → 追加测试用例（测试依赖）
```

不需要人脑追踪，从依赖图自动输出。

### 支柱三：上下文管理（给 AI 读什么）

让 AI 工作时，"把什么放进上下文"极为重要。

- 把全部源码放进去 → Token 数超限，或遗忘中间部分（Lost in the Middle 问题）
- 太少 → 遗漏影响位置

CoDD 从依赖图中动态选取"当前此次变更所需的信息"传给 AI，并根据重要程度以全文/摘要/仅元信息等不同粒度传递。

### 三大支柱的关系

```
支柱一：需求 → 给出设计候选 → 人类选择
支柱二：通过依赖图双向自动传播变更点  ★核心
支柱三：从依赖图选取必要信息传给 AI
```

支柱二是核心。支柱一和支柱三都依赖于支柱二的依赖图才能运作。

## 什么是依赖图

### 基础

图论中的图。由**节点（顶点）**和**边（连线）**构成。

```
[A] ---调用--→ [B] ---读取--→ [C]
```

- 节点 = 事物（文件、函数、配置键、DB 表、设计文档的章节，等等）
- 边 = 事物之间的关系（A 调用 B，B 读取 C，等等）

### 普通依赖图（现有技术）

和 IDE 的"查找引用"做的事情一样。节点是源代码元素，边是 `imports`、`calls`、`extends` 这类代码层面的关系。可以通过 AST（抽象语法树）或 LSP（Language Server Protocol）自动检测。

### CoDD 的依赖图：有什么不同

普通依赖图只在代码内部。CoDD 在其中**混入了代码以外的事物**。

**节点种类更多：**
```
代码：      processor.py, stripe_adapter.py, charge()
配置：      config.yaml 中的 payment.provider 键
DB 表：     payment_fees 主数据
设计文档：  payment_flow.md 第 3 节
测试：      test_processor.py
批处理：    nightly_settlement.py（夜间批处理）
隐性规则：  "adapter 之间必须保持接口一致"这一惯例
```

不同种类的事物共存于同一张图中。代码、DB、设计文档、测试和隐性知识，在同一张图中通过边连接。

**边的种类也更多：**
```
imports      — 导入文件（自动检测）
calls        — 调用函数（自动检测）
reads_config — 读取配置值（可自动检测）
switched_by  — 由配置值切换调用目标（LLM 推断 or 框架分析）
driven_by    — 主数据控制行为（LLM 推断 + 人工注解）
convention   — 按惯例应一起修改（人工告知 or git 历史）
```

前三种可以机械地提取。后三种光看代码是看不出来的。**这就是 CoDD 的前沿。**

### 具体案例：支付处理模块

```
[processor.py::route_payment]
    │
    ├── reads_config ──→ [config.yaml::payment.provider]
    │                        │
    │                        ├── switched_by ──→ [stripe_adapter.py]
    │                        │                    （provider=stripe 时）
    │                        │
    │                        └── switched_by ──→ [paypal_adapter.py]
    │                                            （provider=paypal 时）
    │
    ├── driven_by ────→ [DB: payment_fees 主数据]
    │                    （手续费率控制逻辑）
    │
    └── implements ───→ [设计文档: payment_flow.md 第 3 节]

[stripe_adapter.py]
    │
    └── convention ───→ [paypal_adapter.py]
                        （接口保持一致的惯例）
```

这就是"资深工程师脑海中"的可视化呈现。20 年老将对这一结构心知肚明。CoDD 将其转化为明确的数据结构。

## Q&A：关于依赖图的常见问题

### Q：数据怎么存储？YAML 文件？DB？

**分阶段而定。**

| 阶段 | 存储形式 | 原因 |
|------|---------|------|
| 初期（小~中规模） | YAML + SQLite | 可手动编辑，可用 git 管理差异，轻量 |
| 规模变大后 | SQLite（本体）+ Neo4j（搜索用） | 节点数万、边数十万时 YAML 已无法胜任 |

**YAML 示例：**
```yaml
nodes:
  - id: src/payment/processor.py::route_payment
    type: method
    description: "支付路由"

  - id: config/payment.yaml::provider
    type: config_key
    possible_values: [stripe, paypal, internal]

edges:
  - from: src/payment/processor.py::route_payment
    to: config/payment.yaml::provider
    type: reads_config
    confidence: 1.0
    source: auto_detected

  - from: config/payment.yaml::provider
    to: src/payment/stripe_adapter.py
    type: switched_by
    confidence: 0.95
    condition: "provider == 'stripe'"
    source: llm_inferred
```

YAML 的优势是人类可读、可写、可修改、可用 git 管理差异。YAML 的局限是节点超过 1 万个时搜索会变慢。

中小规模项目从 YAML 开始，规模变大后迁移到 SQLite 就足够了。

使用场景汇总：
```
YAML   ──→ 人工编写的注解、惯例规则、手动定义（入口）
SQLite ──→ 图的本体。供机器高速检索（本体）
Neo4j  ───→ 路径搜索、可视化。仅大规模时追加（可选）
JSON   ────→ 传递给 AI Agent（轻量切片）
```

### Q：边上的"置信度"和"条件"是什么？

CoDD 的边不只是一条线，还附带三个重要属性。

**属性一：置信度（confidence）——可信程度**

```
imports（代码导入了）          → 1.0（确定）
calls（调用了函数）            → 1.0（确定）
switched_by（由配置值切换）    → 0.95（基本确定）
driven_by（主数据控制）        → 0.8（比较确定）
convention（按惯例一起修改）    → 0.7（大概是这样）
```

可自动检测的置信度高；越依赖人工知识或 LLM 推断的置信度越低。

**属性二：证据来源（Evidence Ledger）——为什么说这条边存在**

同一条边可以附带多条证据。

```
stripe_adapter → paypal_adapter 的惯例边：
  证据 1：git 历史中 12 次有 11 次同时修改（history, score: 0.75）
  证据 2：PR 评审中有"要保持接口一致"的评论（human, score: 0.90）
  证据 3：LLM 推断"因为接口相同"（inferred, score: 0.60）
```

置信度从证据台账中计算得出，而非直接硬写，保持可从依据重新计算的状态。多条弱证据汇聚时置信度上升（Noisy-OR 机制）。这是对资深人员"几个迹象吻合就觉得可疑"这一直觉的建模。

**属性三：有效条件（condition predicate）——这条边什么时候有效**

```yaml
- from: config/payment.yaml::provider
  to: src/payment/stripe_adapter.py
  type: switched_by
  condition: "provider == 'stripe'"
```

生产环境中 `provider=stripe`，所以这条边有效。但开发环境可能是 `provider=internal`，那么这条边在该环境中就无关了。

没有条件，所有环境的依赖就会混在一起，产生"和生产环境无关却提示有影响"的噪声爆炸。资深工程师在脑中就会自动过滤条件，CoDD 将这一过程明确地作为数据来保存。

### Q：这个依赖图怎么构建？要手动写吗？

**分阶段自动构建。全部手动写是不可能的。**

#### Phase 1：机器可提取的自动提取（数分钟到数十分钟）

用 AST 解析工具（Tree-sitter 等）机械地分析代码。

```python
# 分析以下代码：
from stripe_adapter import StripeAdapter

def route_payment(amount):
    provider = get_config('payment.provider')
    if provider == 'stripe':
        return StripeAdapter().charge(amount)
```

```
自动检测到：
  processor.py ── imports ──→ stripe_adapter.py （置信度: 1.0）
  route_payment ── calls ──→ StripeAdapter.charge （置信度: 1.0）
  route_payment ── reads_config ──→ payment.provider （置信度: 1.0）
```

无需人工或 LLM，AST 解析器只解析语法。**仅此一步就能覆盖 60~70% 的依赖。**

#### Phase 2：框架分析 + LLM 推断（数小时）

以 Phase 1 的 `reads_config`（"读取了配置"）为起点，深入挖掘**其后的依赖**。

**先向框架询问（以 Spring Boot 为例）：**

通过 Spring Boot Actuator 这一框架标准机制，可以获取 DI 中各组件的连接关系。

```
GET /actuator/beans → 输出已注册的 Bean 列表及其依赖关系
GET /actuator/conditions → 输出 @Profile 和 @ConditionalOnProperty 的评估结果
```

无需 LLM，即可获取配置与实现的对应关系。最初的设计忽略了这一点，以为"只能让 LLM 推断"，但实际上向框架询问可以获取相当多的信息。

**只将框架无法提取的部分交给 LLM：**

对于自定义工厂或独特的注册表模式等机械分析无法处理的内容，让 LLM 阅读后给出候选。但 LLM 做的不是"凭空想象依赖"，而是"对模式进行分类并给出候选"。

**LLM 是最后手段。** 机器可提取的 → 从框架可提取的 → 剩下不清楚的部分才交给 LLM。按这个顺序来。

#### Phase 3：向人工确认（持续进行）

将 LLM 的推断结果展示给资深工程师，引导他们输出隐性知识。

```
CoDD：   "我们推断 stripe_adapter 和 paypal_adapter
          通过 config.payment.provider 切换。
          这样理解正确吗？"

资深人员："正确。另外还有 internal_adapter。
          还有，这三个接口必须保持一致。"

→ 添加节点 + 添加 convention 边
```

从零开始问"请把所有依赖关系都写下来"，没有人会去做。但"这样理解对吗？还有别的吗？"就能得到答案。以 LLM 的推断为种子，引出人类的隐性知识。

#### Phase 4：从历史数据学习（持续、自动）

| 数据来源 | 可获知的内容 | 生成的边 |
|----------|------------|---------|
| git log | 同时被修改的文件对 | co_changed_with |
| 故障工单 | "改了 X 却忘改 Y 引发故障" | incident_correlated_with |
| 代码评审 | "这里也要修改"的评论 | convention_requires |
| 测试失败日志 | X 的变更导致 Y 的测试损坏 | validated_by_test |

随着时间推移，依赖图会越来越"聪明"。最初的心智模型较弱，但持续使用后会逐步接近资深人员的水平。

### Q：如果 SQL 是动态生成的怎么办？

企业系统中的常见情况。SQL 通过字符串拼接或 ORM 的方法链动态组装。

```java
// 静态分析可追踪
String sql = "SELECT fee_rate FROM payment_fees WHERE provider_code = ?";

// 静态分析无法追踪
String sql = "SELECT " + columnName + " FROM " + tableName;

// 更无法追踪
StringBuilder sql = new StringBuilder("SELECT fee_rate FROM payment_fees");
if (hasDiscount) {
    sql.append(" JOIN discount_master ON ...");
}
```

**三步应对：**

**第一步：能静态获取的先静态获取。** 直接的 SQL 语句、ORM 注解（`@Query` 等）、Hibernate 或 MyBatis 的映射定义。这样能覆盖六七成。

**第二步：实际运行并观测（runtime observation）。** 动态 SQL 一旦执行就能确定。用 OTel（OpenTelemetry）获取 DB Trace，就能记录"实际读取了哪个表的哪个列"。观测比推断更可靠。

```
Trace 记录示例：
  span: db-query
    sql: "SELECT fee_rate FROM payment_fees
          JOIN discount_master ON ...
          WHERE provider_code = 'stripe'"
    tables: [payment_fees, discount_master]  ← 发现了 discount_master！
```

**第三步：LLM + 人工。** 对于仍然无法获取的特殊情况（如只在年末运行的批处理等），让 LLM 做模式分析，再向资深人员确认。

实际比例参考：
```
静态分析可获取：     60-70%
runtime 观测可获取：  15-20%
LLM 推断可获取：      5-10%
人工告知：            5-10%
无法获取：            数%（未观测的罕见路径）
```

100% 不可能做到，但获取 90% 在实用上已经足够。

### Q：runtime 观测的结果怎么反馈到依赖图中？

在应用中接入 OTel（OpenTelemetry）。应用正常运行时会记录 Trace。

```
生产/预发布环境（正常运行中）
    │
    │  每次执行都会记录 Trace
    │  "哪个函数发出了哪条 SQL，
    │    读取了哪个表，之后又调用了什么"
    │
    ▼
反馈管道（夜间批处理等）
    │
    │  分析 Trace 数据并更新依赖图：
    │  - 已知的边 → 追加证据，置信度提升
    │  - 未知的边 → 新建（source: dynamic）
    │
    ▼
依赖图更新完成
```

具体案例：静态分析只检测到了 `payment_fees` 表，但 runtime 观测发现 `discount_master` 表也被 JOIN 了。

```
改造前的依赖图（仅静态分析）：
  StripeAdapter.charge → payment_fees     ← 已知

改造后的依赖图（runtime 观测后）：
  StripeAdapter.charge → payment_fees     ← 已知
  StripeAdapter.charge → discount_master  ← ★新发现！
```

下次有人修改 StripeAdapter 时，discount_master 也会出现在影响列表中。

无需额外操作，只需在应用中接入 OTel 正常运行即可。观测次数越多，证据积累越多，置信度越高。

### Q：是要在改造前先把所有源码的图建好吗？新项目开发时不能用吗？

**新项目开发和改造的用法不同。**

| 阶段 | CoDD 的用法 |
|------|-----------|
| 新项目开发（从零开始） | 支柱一（设计生成）为主角。依赖图在 CI 中后台积累 |
| 成长期（功能追加增多） | 依赖图开始发挥作用，投入传播引擎 |
| 维护、改造 | **CoDD 的本领所在。** 全功能全速运转 |

新项目开发时用敏捷快速迭代，每天代码都在剧变，看依赖图也没有意义。但**只需在 CI 中接入 Phase 1（静态分析）即可**。每次提交时图会自动更新，平时用不上但在后台积累。

v1.0 稳定后依赖图开始有意义。进入运营维护阶段后可以充分发挥。

**CoDD 的核心是"用于改造的工具"。** 最初提到的"企业级大型系统中改了这里哪里会崩"是核心问题，因此在改造阶段自然最为有效。

### Q：不同场景下用法不一样吗？

确实如此。用法在四种场景下各有不同。

**场景一：新项目开发**
- 支柱一（设计生成）为主角，依赖图在 CI 中后台积累
- Day 1 看依赖图没有意义，是为将来播下的种子

**场景二：遗留系统维护**
- CoDD 最有效。防止"改了这里那里崩了"的漏检
- 最优先的是在资深人员离职前完成 Phase 3（知识提取）
- 不要一次性铺开，从修改频率×故障频率高的区域开始

**场景三：现代化改造（遗留→现代迁移）**
- 可以用依赖图制定"哪里可以先切离"的迁移计划
- 重点管理新旧之间共享 DB、共享 API 的边
- 与 Strangler Pattern 配合良好。指向旧节点的边为零 = 可安全删除

**场景四：现有系统增强（★最常见的情况）**
- "加入新功能"本身不难，"保证现有功能不被破坏"才难
- CoDD 的价值在于**设计前的影响范围预测**。不是做完了才发现崩了，而是在设计阶段就能比较
- 可以从图中定量比较设计方案 A vs 方案 B 的影响范围（Amber 区的数量）

增强特有的用法：

| 用法 | 维护 | 增强 |
|------|------|------|
| 影响分析 | 变更后"哪里崩了？" | 变更前"哪里可能崩？" |
| 设计决策 | 不需要（直接修就行） | 方案 A vs 方案 B 的影响范围比较 |
| 回归测试 | 崩了的地方的测试 | 从依赖图自动提取影响测试 |

对回归测试计划也有效：
```
传统方式："为了保险起见，把所有测试都跑一遍" → 3 天
CoDD：    自动提取受影响的测试 → Amber 区 50 件 + Gray 区 30 件
          无影响的 500 件可跳过 → 3 天缩短为半天
```

### Q：还没成长起来的 CoDD（刚导入时）有用吗？

Day 1 不会发生魔法。成长曲线如下：

```
Day 1：   仅 Phase 1。与 grep + IDE 搜索差异不大，但已结构化为图
Week 1：  Phase 2 也跑起来。配置驱动的依赖开始可见，超越了 grep
Month 1： Phase 3 录入了资深人员的知识。惯例、隐性知识上了图
Month 3： Phase 4 开始运转。作为"弱心智模型"达到实用水平
```

还没成长起来也可以接入 CI 让它自然成长。日常工作（提交、评审、测试、故障处理）都会成为培育图的数据。

最大的风险是"资深人员离职后才慌忙应对"。还没成长起来，也应尽早启动 Phase 3（向人工确认）。离职后就问不到了。

## 变更点传播：变更发生后会怎样

### 基本运作

```
1. 发生变更（例：charge() 的签名变更）
2. 从变更点出发，沿边走向相邻节点
3. 从相邻节点继续追踪
4. 每追踪一步，"可能受影响的程度"分数下降
5. 分数低于阈值时停止
6. 输出所有追踪到的节点作为"影响列表"
```

就像向水面投石时扩散的涟漪。以变更点为中心波纹向外扩散，越远越弱。

### "变更了什么"决定了波的扩散方式

即使是同一个文件的变更，种类不同，传播模式也截然不同。

| 变更种类 | 波的扩散 | 原因 |
|---------|---------|------|
| public 方法签名变更 | 范围广 | 所有调用方都受影响 |
| 仅内部实现变更 | 范围窄 | 对外暴露的行为不变 |
| 重构（无语义变更） | 几乎不传播 | 只是改了名字 |
| 配置默认值变更 | 有条件地范围广 | 只影响使用该配置的环境 |
| 主数据语义变更 | 最危险 | 逻辑分支会改变 |
| 仅文档修改 | 不传播 | 对代码无影响 |

不区分变更种类一律传播，会产生大量噪声，变得无法使用。

### 如何停止传播

涟漪无限扩散会导致所有节点都显示"受影响"，失去意义。停止传播有五个标准。

```
① 在契约边界停止（最优先）
   public API 的契约本身没有变化，就不向外传播

② 条件不成立
   生产环境只用 Stripe → paypal 的依赖在生产环境中不传播

③ 没有增加新信息
   只会输出同一类影响时停止

④ 风险分数过低
   越远分数越低

⑤ 深度限制（最后的安全阀）
```

### 影响列表的呈现方式：三带分类

将影响列表按重要度分为三个区段。

**Green（可自动更新）：**
```
1. [auto] test_processor.py — 跟随参数变更
2. [auto] docs/api/payment.md — 更新 API 签名
```
置信度极高，机械地就能确定正确答案。由 AI 修改，人类只需确认 diff。

**Amber（必须人工评审）：**
```
3. [review] stripe_adapter.py — 接口一致性
4. [review] paypal_adapter.py — 同上（★仅影响生产 paypal 环境）
5. [review] DB: payment_fees — 与手续费率主数据的一致性
```
一旦遗漏就会引发故障的位置。即使有些噪声，也以不遗漏为优先。

**Gray（参考信息）：**
```
6. [info] nightly_settlement.py — 对夜间批处理的影响
7. [info] monthly_report.py — 月度报表
```
可能有关系的程度。有余力时再看。

根据产出物种类，阈值有所不同。费率计算代码，稍有可疑就列入 Amber；README，非重大情况一般列为 Gray。

## Q&A：关于传播引擎的常见问题

### Q：会自动帮我修好所有地方吗？还是只告诉我"看这里"？

**因区段而异。**

| 区段 | AI 的角色 | 人类的角色 |
|------|---------|---------|
| Green | **AI 来修** | 确认 diff 并点击 OK |
| Amber | **AI 给出多个修改方案** | 选择采用哪个 |
| Gray | **AI 告知确认要点** | 自己阅读后判断 |

Amber 区段示例：
```
CoDD："charge() 的签名发生了变化。
       paypal_adapter.py 也会受到影响。

       修改方案 A：修改为相同签名
       修改方案 B：在 processor.py 的路由侧吸收

       → 请选择采用哪个？[A/B/手动提供其他方案]"
```

不是全靠人工，也不是全自动。AI 和人类的分工随区段而变化。

"全自动修复"是最终目标。最初作为"防止遗漏的安全网"提供。随着信任积累，逐步扩大 Green 区段（自动更新的范围）。

### Q：Green 区段最初很少，会逐渐增多吗？为什么？

**团队正常工作，证据就会积累，置信度就会提升。**

最初（仅 LLM 推断）：
```
stripe_adapter → paypal_adapter 的惯例边：
  证据 1：LLM 推断（score: 0.60）
  → 置信度: 0.60 → Amber（人工评审）
```

3 个月后（证据已积累）：
```
stripe_adapter → paypal_adapter 的惯例边：
  证据 1：LLM 推断（score: 0.60）
  证据 2：git 历史中 12 次有 11 次同时修改（score: 0.75）
  证据 3：评审中有"接口要保持一致"的评论（score: 0.90）
  证据 4：只改了一个，集成测试就失败了（score: 0.95）
  → 置信度: 0.9995 → Green（可自动更新）
```

提交代码、做评审、测试失败、发生故障。这些全部都在为边添加证据。不需要特别的作业，日常工作就在培育图。

反之，如果对代码进行大规模重构，已有证据会变旧，置信度下降，自动从 Green 退回 Amber。具有自净作用。

### Q：与其是个人的知识，不如由组织来共同培育，不是吗？

**正是如此。这也是 CoDD 本质价值之一。**

现实情况：
```
田中（20 年老将）脑中："改了这里那里要一起改"
佐藤（15 年老将）脑中："那个主数据影响那个批处理"
铃木（10 年老将）脑中："这个配置只在生产环境有效"
→ 全部散落在各人脑中。退职就消失。
```

录入 CoDD 依赖图后：
```
田中告知 → convention 边（evidence: human:tanaka）
佐藤告知 → driven_by 边（evidence: human:sato）
铃木告知 → condition: "env == 'prod'"（evidence: human:suzuki）
→ 个人的隐性知识，转化为组织的共享资产（图）。
→ 记录了是谁告知的。他人可以修改。人员离职后仍然保留。
```

Phase 3（向人工确认）和 Phase 4（从历史学习）本身就是组织学习。每次评审、每次提交、每次故障处理，团队全员的知识都在图中积累。

| 问题 | CoDD 的解决方式 |
|------|-------------|
| 知识依赖个人 | 隐性知识记录在图中，人员离职后仍保留 |
| 知识断层 | 新成员可以通过看图快速上手 |
| 入职培训 | "改了这里看哪里"从系统中输出 |
| 评审质量参差不齐 | 图补充评审视角 |

## 支柱三：上下文管理——给 AI 读什么

### 问题

影响列表已经输出了。想让 AI 对 Amber 区段给出修改方案。但给 AI 读什么，会导致修改质量天差地别。

```
坏例子 1：把全部源码放进去（10 万行）
  → Token 数超限，根本放不下。
  → 就算放下了，也会遗忘中间部分（Lost in the Middle 问题）

坏例子 2：只放修改对象的文件
  → 不知道"这个方法为什么存在"
  → 看不到与配置值或主数据的关系
  → 给出的修改方案牛头不对马嘴
```

### Lost in the Middle 问题

LLM（大型语言模型）的注意力机制的弱点。

```
上下文窗口：
┌──────────────────────────────────────┐
│ [开头] ← 记得好                        │
│                                      │
│ [中间] ← 容易遗忘 ⚠️                   │
│                                      │
│ [末尾] ← 记得好                        │
└──────────────────────────────────────┘
```

开头和末尾记得好，但放在中间的信息容易被遗忘。所以如果把重要代码放在中间，AI 就会忽略它来给出修改方案。

### CoDD 的解决方案：依赖图自动构建上下文

因为有依赖图，"当前此次变更涉及什么"就很清楚了。利用这一点自动选择传给 AI 的信息，并以最优顺序排列。

```
上下文窗口的构成：

┌──────────────────────────────────────┐
│ [开头：记得好]                          │
│                                      │
│  1. 修改对象文件（全文）                 │
│  2. 变更的意图、目的                    │
│  3. 直接相关的接口（全文）               │
│                                      │
├──────────────────────────────────────┤
│ [中间：容易遗忘 → 所以只放元信息]          │
│                                      │
│  4. 间接相关的模块（仅摘要）             │
│     "这个模块执行○○"程度就够            │
│  5. 配置值列表及当前值                  │
│  6. 主数据的结构（仅 DDL）              │
│                                      │
│  ※ 不放代码全文。只放就算忘了           │
│    也不会造成致命影响的信息              │
│                                      │
├──────────────────────────────────────┤
│ [末尾：记得好]                          │
│                                      │
│  7. 影响集合的汇总表                    │
│  8. 过去的相关故障信息                  │
│  9. 检查清单                           │
│     "请确认以下内容"                   │
│                                      │
└──────────────────────────────────────┘
```

三个关键点：

**关键点一：开头放修改对象，末尾放检查清单。** 将重要信息放在注意力高的位置。

**关键点二：中间不放代码全文。** 只放元信息（"这个模块执行支付处理"程度的摘要），就算忘了也不会造成致命影响。

**关键点三：需要时再即时读入。** AI 看到中间的元信息后判断"需要更详细地了解这里"，则在该时刻读入该文件全文。不要从一开始就全部放进去。

### "放什么进去"的判断标准

不是看重不重要，而是看**是否模糊**。

```
重要但不模糊 → 摘要就足够
  例："这个模块调用 Stripe API"（不言而喻）

重要且模糊 → 读入全文
  例："这个模块的分支条件不明"

不重要 → 只放元信息，或不放
```

很多失败源于"因为重要所以放全文"的思路。实际上"因为模糊所以需要全文"。重要但不言而喻的东西，摘要就足够了。

### 与依赖图的联动

```
1. 传播引擎输出影响列表
2. 对各节点判断应该放"全文/摘录/摘要/元信息"的哪一种
3. 在 Token 预算内选取最有用的组合
4. 以 U 字形排列（重要内容放开头和末尾）
5. 传给 AI
6. AI 判断"需要这里的全文"时追加读入
```

没有依赖图，人类每次都要手动判断放什么进去。有了依赖图，就能自动决定"这次变更需要这些信息"。这就是支柱二与支柱三的联动。

## 三大支柱构成一个循环

三大支柱不是独立的功能，全部相互连接，循环运转。

### 整体流程

```
【支柱一】收到需求
  │
  │  "为支付功能追加多货币支持"
  │
  ▼
  AI 针对依赖图生成多个"变更方案"
  │
  │  方案 A：追加 currency 列 + 修改 adapter + 追加主数据
  │  方案 B：新建货币转换层 + 不动现有代码
  │
  ▼
  人类选择 → 采用方案 A
  │
  │  → 此时"什么会发生变化"已在图上确定
  │
  ▼
【支柱二】通过依赖图传播影响
  │
  │  选择了方案 A 后：
  │  - payment_fees 追加 currency 列（SchemaChanged）
  │  - processor.py 参数变更（InterfaceContractChanged）
  │  - stripe/paypal 两个 adapter 受影响（经 switched_by）
  │  - 设计文档第 3 节受影响（经 implements）
  │  - 测试规格追加多货币测试（经 validated_by）
  │  - 夜间批处理可能受影响（经 co_changed_with）
  │
  │  → 分类为 Green / Amber / Gray
  │
  ▼
【支柱三】向 AI 传递最优上下文
  │
  │  为修改 Amber 区的 stripe_adapter：
  │  - 开头：stripe_adapter.py 全文 + processor.py 的接口
  │  - 中间：paypal_adapter 的摘要 + payment_fees 的 DDL
  │  - 末尾："统一 currency 的处理方式"检查清单
  │
  ▼
  AI 给出修改方案
  │
  │  "stripe_adapter.py 这样修改如何？（方案 A/方案 B）"
  │
  ▼
  人类评审 → 采用 → 提交
  │
  ▼
【回到支柱二】提交后依赖图自动更新
  │
  │  Phase 1 运行 → 追加/更新新的边
  │  证据增加一条 → 置信度略有提升
  │
  ▼
  下次变更时，以更高精度的状态进行传播
```

### 循环图

```
支柱一（设计生成）──→ 支柱二（影响传播）──→ 支柱三（上下文优化）
      ↑                                          │
      │                                          │
      └──────── 反馈 ──────────────────────────────┘
                （修改结果培育图）
```

- 支柱一决定"改什么"
- 支柱二输出"影响哪里"
- 支柱三决定"给 AI 读什么"
- AI 给出修改方案 → 人类确认 → 提交
- 提交培育支柱二的图
- 下一个周期精度更高

**转得越多越聪明。三大支柱不是分散的功能，而是一个循环系统。**

## CoDD 的本质：组织知识与经验的结晶

从技术上说是"变更影响分析工具"，但本质不止于此。

```
依赖图 = 源代码的结构（机器可读的部分）
       + 框架的配置信息（机器可查询的部分）
       + 运行时行为（机器可观测的部分）
       + 资深人员的经验知识（从人类引出的部分）
       + 团队的历史（从 git / 故障 / 评审中学到的部分）
```

也就是说，**"所有参与过该系统的人的知识与经验"结晶在图中。**

### 为什么传统的知识管理会失败

| 现有方法 | 问题 |
|---------|------|
| 文档化（Confluence/Notion） | 不写。就算写了也不更新，会腐烂 |
| 交接资料 | 离职前才慌忙写，漏洞百出 |
| 结对编程/集体编程 | 知识可转移但不被记录 |
| 代码注释 | 不写。就算写了也只有"是什么"，没有"为什么" |

### CoDD 的依赖图为何不同

CoDD 的依赖图不是"特意去写"，而是"作为日常工作的副产品自然成长"。

- CI 每次提交自动更新（不会腐烂）
- runtime 每天观测（自动成长）
- 每次评审都录入人工知识（边用边成长）
- 发生故障后自动追加证据（痛苦的经历转化为知识）
- "为什么"得以保留——证据（Evidence）中记录了"上次故障""评审评论"
- 人员离职后仍然保留——已结晶在图中

### CoDD 一句话概括

```
技术层面的描述：
  "基于条件证据图的变更影响分析"

打动人心的描述：
  "团队全员的知识与经验，结晶在越用越聪明的依赖图中。
   人员离职，知识留存。"
```

## CoDD 的演进方向：AI 自动修复与半自动增强

依赖图越成熟，AI 可自动化的范围就越广。

### Level 1：告知影响（初始阶段）
```
人类："要改这里"
CoDD："这里也受影响"（Green/Amber/Gray）
人类：自己修改，或从 AI 的修改方案中选择
```

### Level 2：修改几乎全自动（扩大 Green 区段）
```
人类："要改这里"
CoDD：识别影响 + 自动修复 Green 区 + 展示 Amber 区的修改方案
人类：只需对 Amber 区进行判断
```

### Level 3：自动修复故障
```
监控："生产环境出现错误"
CoDD：定位错误位置 → 通过依赖图识别原因候选
      → "最近的 commit X 是原因的可能性 95%"
      → 自动生成修复补丁 → 自动运行测试 → 通过
人类：评审补丁并合并即可
```
因为依赖图中记录了"此次 commit 改变了哪条边"，从错误位置反向追踪，就能定位到最近变更的边。

### Level 4：半自动增强
```
PM："帮我加订阅功能"
CoDD：解析需求 → 生成多个变更方案 → 比较影响范围
      → 推荐风险最低的方案
      → 生成代码、测试、设计文档、migration
人类：只做设计决策 + 最终评审
```

### 为什么有 CoDD 才能做到这些

与普通 AI 工具（Cursor、Copilot 等）的区别：

| 项目 | 普通 AI | 搭载 CoDD 的 AI |
|------|---------|----------------|
| 上下文 | 当前打开的文件及周边 | 从依赖图动态构建 |
| 全系统影响 | 不知道 | 知道 |
| 配置驱动的依赖 | 看不见 | 看得见 |
| 过去的故障模式 | 不知道 | 知道 |
| 资深人员的知识 | 没有 | 结晶在图中 |

AI 的能力相同，给予的上下文质量差了不止一个量级。

### 到达条件

Level 1-2 马上就能做到。Level 3-4 需要的条件：
- 足够成熟的依赖图（Month 3 以上）
- 较高的测试覆盖率
- 完善的 CI/CD 流水线
- Green 区的精准率在 95% 以上

只要持续做 Level 1-2，就会自动向 Level 3-4 迈进。日常工作培育图，精度提高，自动化范围扩大。

## CoDD 的局限性（如实说明）

### 原理上无法解决的问题

1. **未被记录的意图无法复原。** 没有留在代码、文档或 Trace 中的业务意图，能力有限
2. **罕见路径难以观测。** 一年才运行一次的批处理，观测数据无法积累
3. **历史数据也会学习不良惯例。** 过去夹带 bug 一起 co-change 的模式也会被记住
4. **100% 不可能。** "存在未知依赖的可能性"始终存在

### 在实务中有多大影响？（来自 Q&A）

> **Q：局限性 1"意图消失"，但分析代码后结构还在，依赖不是还能追踪吗？**

确实如此。"为什么这样做的原因"消失了，但"改了这里会影响那里"这一结构性依赖通过 AST 分析仍然可见。用于影响分析已经足够。局限性 1 是"无法解释意图"，而不是"看不到依赖"。

> **Q：说罕见路径无法观测，但罕见路径本来就不会经常修改吧？**

确实如此。年度批处理几乎不会日常改造。唯一的例外是现代化改造（全系统替换）时。那时可以有意地让该路径运行一次并获取 Trace。

> **Q：这些局限性到底是致命的吗？**

不是致命的。哪一条都不会导致"所以 CoDD 没法用"。本节**在很大程度上是论文的惯例写法**。在学术论文中，写"我们的方法万能"会被同行评审否决，因此需要主动说明弱点。"原理上这里做不到，但实务上不太会成为问题"这样的写法，反而会提升可信度。

### 针对局限性 3"学习不良惯例"的对策

> **Q：历史数据学了错误的依赖，有什么应对方法吗？**

有三种对策。

**对策一：与 bug 修复提交做交叉核对。** A 和 B 的 co-change 之后，频繁出现将 B 回滚的 revert 或 bug 修复 commit，则判定"A→B 的 co-change 很可能是 bug"，降低置信度。

**对策二：Evidence Ledger 的多数投票。** 即便基于历史的证据说"A→B"，若静态分析、框架分析、执行 Trace 都未能佐证 A→B，用 Noisy-OR 计算后置信度仍然偏低。只有历史数据支持的依赖，会自动被分类为 Amber（需确认）。

**对策三：人工反馈覆盖。** Phase 3 中人工标记"这个依赖是错的"，就会作为 negative evidence（否定证据）记录到 Evidence Ledger 中。此后同样的模式出现，置信度也会降低。

无法完全防止，但能防止"不良学习失控"。**设计上保证了仅靠历史证据的单一依据不会进入 Green 区段。**

### 因此 CoDD 不追求完美

目标是"覆盖资深人员七成的能力"。剩下的三成靠：
- 测试（变更后如果崩了就能发现）
- 监控（生产环境出现异常就能发现）
- 人工评审（最后的防线）

来补充。CoDD 是**安全网**，而不是银弹。

## CoDD 不是"以 AI 为中心"，而是"让 AI 发挥作用的基础设施"

> **Q：CoDD 完全是以 AI 为中心的开发方法，不是吗？**

乍看如此，但实际不是。

支柱一（依赖图）和支柱二（传播引擎）是**完全确定性的**。AST 分析、静态分析、框架分析、SQL 解析器。没有使用 AI。这是基础。

CoDD 的结构：
- **底层**：确定性的依赖图 + 传播算法（零 AI）
- **中间层**：AI 补充"不足的部分"（Phase 2 的 LLM 分类，Layer 3-5 的推断）
- **输出层**：AI "用结果来工作"（上下文优化，自动修复）

AI 不在中心，而是**为 AI 能够工作而整理数据的基础设施**。

### 与现有 AI 开发工具的区别

现在的 Claude Code 或 Copilot 每次都从零开始阅读代码库并推测依赖关系，就像"每天早上都失忆的资深工程师"。

| 项目 | 没有 CoDD | 有 CoDD |
|------|---------|---------|
| 依赖掌握 | AI 每次推测 | 参照预先计算好的图 |
| 影响范围 | AI"可能这里也有" | 附带证据的"这里，原因是这个" |
| 上下文 | 全部塞进去或凭感觉选 | 只选必要内容并最优排列 |
| 修改建议 | 只看代码片段后提建议 | 看上下游后提建议 |

**结论**：CoDD 不是"以 AI 为中心的开发方法"，而是**"让传统开发具备 AI-ready 能力的基础设施"**。不是以 AI 为中心，而是将系统知识结构化，让 AI 能最大限度发挥能力。主角是"结构化的依赖知识"，AI 是使用它的消费者。

现有 AI 开发工具（Copilot、Cursor、Claude Code）的方向是"让 AI 更聪明"，CoDD 的方向是"让喂给 AI 的数据更聪明"。这是决定性的区别。

### 主体的逆转：进入人类为 AI 提供支撑的时代

> **Q：这不就是"AI 辅助人类"颠倒成了"人类辅助 AI"吗？**

确实如此。传统的 AI 开发辅助是"人类设计→人类编码→AI 补全"。主体是人类，AI 是助手。

当 CoDD 演进到 Level 3-4，就变成了"AI 识别影响范围→AI 生成修改方案→人类审批"。**主体变成了 AI，人类成了"确认是否正确"的评审者。**

这是 paradigm shift（范式转移）。GPT-5.4 将 CoDD 评价为"适合 vision paper"，正是因为它在探讨这样一种方向性的重大转变。

### 迈向"AI 驱动组织"的完美基础

2025-2026 年的世界趋势是"AI 驱动组织""AI-native 企业"——AI 为主体，人类作补充的工作方式。但"在软件开发中具体如何实现"的方法论，没有人给出答案。

现有的 AI 开发工具（Copilot、Cursor、Claude Code、Devin）全都是"人类→AI"方向。AI 主动做出判断的知识基础并不存在。所以每次都需要人类来告知依赖关系。

CoDD 给出了答案。为 AI 自主做出开发决策，提供结构化的知识基础。

- 更聪明的 AI → 等待模型进化（CoDD 的范畴之外）
- 更好的 Prompt → 仍然依赖人类（无法规模化）
- **AI 可参照的结构化知识** → CoDD 提供的东西

**CoDD 是实现"AI 驱动开发"的具体方法论。** 其价值不在于技术的新颖性，而在于"问题的提法本身是新的"。

vision paper 的核心论点："While current AI-assisted development tools augment human developers, the emerging paradigm of AI-driven organizations demands the inverse: structured knowledge infrastructure that enables AI to lead development decisions, with humans providing oversight."

### 超越软件开发的普适性

> **Q：CoDD 不只用于开发，还能应用于整个组织活动吗？**

可以。将 CoDD 解决的问题抽象化，就是"在复杂系统中，当某个元素发生变化时，其他哪里会受到影响"。这不仅限于代码。

组织中发生同样问题的例子：
- **人事异动**：A 离开后，B 项目的隐性联络窗口消失 → 交接遗漏
- **内部规程变更**：修改费用报销规程，还需要修改审批流程和财务系统的配置
- **法规应对**：发票制度 → 发票、会计分录、交易方主数据、合同模板，全部受影响
- **组织改编**：部门合并 → 权限表、邮件列表、审批路线、座位图，全部要改

CoDD 的三大支柱可以直接替换：
- 代码的依赖图 → 业务流程、规程、人员、系统的依赖图
- 条件边 → "仅此部门""仅此合同形态"
- Evidence Ledger → 依据（规程、合同、会议记录）
- 传播引擎 → "改了这里那里也要改"的自动通知
- 上下文优化 → 只向相关负责人发送与其有关的变更

但作为论文策略，先聚焦于软件开发才是正确选择。"什么都能用"会被同行评审批评为"什么都没有证明"。在 cs.SE 建立实绩后再推广到普适场景。

## 两个层次：代码内的传播 与 开发流程间的传播

CoDD 的依赖图有两个层次。

### 层次一：代码内的依赖传播
`Service.java → Repository.java → 测试` 这类系统内部依赖。由支柱二的传播引擎负责。

### 层次二：开发流程产出物间的传播
`需求定义文档 → 概要设计 → 详细设计 → 代码 → 测试规格书 → 运维手册` 这类 V-model 产出物间的依赖。由支柱一的"设计文档⇔代码双向同步"负责。

### 一张图包含全部

CEG 的节点类型涵盖产出物和代码：
- requirement（需求定义的条目）
- design（概要设计、详细设计的章节）
- code（类、方法、配置文件）
- test（测试用例、测试规格书的条目）
- operation（运维手册、监控配置）
- config（application.yml、环境变量）
- data（表定义、主数据）

层次一（代码内）和层次二（流程间）都用**同一张图、同一个传播引擎**处理。

### 双向传播是 CoDD 的真谛

不只从上往下，还可以从下往上传播。`point_master` 追加列 → 详细设计追加列定义 → 概要设计更新 ER 图 → 确认需求定义。以前这些全部由人类在脑中追踪，现在沿图追踪，自动输出"这里也要看"。

## 总结

```
CoDD = 条件证据图（依赖管理）
     + 变更点传播引擎（自动识别影响）
     + 上下文优化（向 AI 选择信息）
```

**提供的内容：**
- "什么可能相关"（影响列表）
- "看到哪里才算安全"（Green/Amber/Gray）
- "为什么这样说"（证据 + 影响路径说明）
- "组织知识的积累"（人员离职后仍留存的知识库）

这就是实务可用的"弱心智模型"。
