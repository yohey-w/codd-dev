# Migrating to CoDD v2.11.0 — Sprint-less `codd implement`

CoDD v2.11.0 removes the Sprint / Wave / `implementation_plan.md` layer from
`codd implement`. The new flow takes a design node and an output path
list directly. There is **no compatibility shim**: legacy invocations now
return a CLI error from v2.11.0 onward.

## Why we removed Sprint

Sprint and Wave existed to slice an implementation plan for human project
management. The CoDD pipeline doesn't need them — when AI generates,
verifies, and repairs in a single loop, the human-process layer just adds
parsing surface and rigid output paths.

Concrete reasons:

1. **`implementation_plan.md` parsing was fragile.** A 110-row table with no
   Sprint heading was being misread as 110 separate tasks.
2. **`output_dir` was hard-coded to `src/generated/sprint_N/`.** Code that
   should land in `tests/` or `docs/` could never get there.
3. **Phase 4 (test) never ran when generate→implement order broke.**
   The Wave dependency made the failure invisible until the missing tests
   surfaced as DAG violations.

## What changed

| Before (≤ v2.10.0) | After (v2.11.0) |
| --- | --- |
| `codd implement [--wave N]` | `codd implement --design <design_doc> --output <dir> [--output <dir>...]` |
| Reads `docs/plan/implementation_plan.md` (or design fallback) | Reads the design doc passed via `--design` |
| Output forced to `src/generated/sprint_N/<task>/` | `--output` may be passed multiple times, each path is project-relative or absolute |
| Sprint / Wave inferred from `m1.x` / `1-x` task IDs | No Sprint or Wave concept; nothing to infer |
| `ImplementationPlan` / `ImplementationTask` dataclasses | `ImplementSpec(design_node, output_paths, dependency_design_nodes)` |

## Migration cookbook

### CLI

```bash
# Before
codd implement --wave 1

# After
codd implement \
    --design docs/design/auth.md \
    --output src/auth/ \
    --output tests/auth/
```

### Programmatic

```python
# Before
from codd.implementer import _load_implementation_plan, _execute_task
plan = _load_implementation_plan(project_root, config)
# ...

# After
from codd.implementer import ImplementSpec, run_implement

spec = ImplementSpec(
    design_node="design/auth.md",
    output_paths=[Path("src/auth/"), Path("tests/auth/")],
    dependency_design_nodes=["design/system.md", "design/database.md"],
)
result = run_implement(project_root, spec, config)
```

### `codd.yaml`

`coherence.implement` and `dag.plan_task_file` are silently ignored. You can
delete them, but you don't have to. If you want to record default
output paths per design, drop them under `implement:` (still optional —
the CLI flags work without it):

```yaml
# Optional in v2.11.0
implement:
  default_output_paths:
    "docs/design/auth.md":
      - "src/auth/"
      - "tests/auth/"
```

### Diff-side findings (`codd diff apply`)

Findings classified as `requirement_only` used to be appended to
`impl_plan.md` / `docs/plan/implementation_plan.md`. They now go into the
project's `docs/requirements/requirements.md` under the same heading
(`## 暗黙要件確認 (codd comparison proposal)`) — colocated with the rest
of the requirements so a future `codd implement --design <path>` picks
up the surrounding section.

### Derived tasks (`codd elicit apply` / plan_deriver)

Approved derived tasks used to be appended to
`docs/design/implementation_plan.md`. They now go into
`docs/requirements/requirements.md` under `## Derived Tasks`, again
colocated with their requirements.

## Upgrade timeline

- **v2.11.0 ships the rewrite immediately.** No deprecation warning, no shim.
  `codd implement --wave N` or running `codd implement` without `--design`
  return a CLI error.
- **Existing `codd.yaml` is still readable.** Legacy `coherence.implement` /
  `dag.plan_task_file` sections are silently ignored, so you can upgrade
  without touching config first.
- **DAG / verify / elicit / diff / brownfield are unaffected** — only
  `codd implement` and the diff-side `_plan_path` routing changed.

## FAQ

**Q. Can I keep using `implementation_plan.md`?**
No. v2.11.0 removes the parser entirely. Either move the content into
your design docs and use `codd implement --design <path>`, or keep the
file as a human-only artifact (CoDD ignores it).

**Q. Is SWE-bench still passing?**
Yes — v2.11.0 release blocker is SWE-bench 73-question 100 % maintained.
If you see regressions on your own benchmark, please file an issue with
the failing case so we can investigate.

**Q. What about backward-compatibility shims?**
There are none, intentionally. The CoDD philosophy is to keep the core
small and unambiguous. A shim layer would double the maintenance surface
without measurable user benefit; the migration above is a single-day
change for the projects we surveyed.
